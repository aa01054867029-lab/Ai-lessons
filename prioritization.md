# Приоритизация фич (MoSCoW)

> Критерии Must have: (1) связана с JTBD primary-персоны, (2) без неё основной сценарий невозможен, (3) table stakes или core-дифференциатор

---

## Must Have — без этого продукт не работает

| # | Фича | Обоснование |
|---|---|---|
| F01 | **KYC-верификация документов** | Table stakes. Без верификации паспорта РФ платформа бесполезна для CO |
| F02 | **Проверка по перечню РФМ** | Core requirement 115-ФЗ. Без этого — штраф до 1 млн руб. Главный дифференциатор vs. глобальные конкуренты |
| F04 | **Проверка паспорта по базе МВД** | Базовый шаг KYC. CO проверяет каждого клиента — entry point в продукт |
| F07 | **Мониторинг транзакций по правилам** | Core сценарий Марины: 40-60 алертов/день. Без мониторинга — нет alert queue |
| F08 | **Alert queue с приоритизацией** | Основной рабочий экран CO. JTBD Марины: «обрабатывать 50-100 алертов/день». Table stakes (Unit21, ComplyAdvantage) |
| F09 | **Bulk-действия с алертами** | При 40-60 алертов/день без bulk — невозможно работать. Марина: «30 ложных × 3 мин = полтора часа» |
| F13 | **Maker-Checker approval flow** | **Core-дифференциатор #1**. Нет ни у одного конкурента. JTBD Алексея: «принять решение по эскалации за 5 мин» |
| F15 | **Эскалация к CEO + override** | Неразрывная часть approval workflow. Без override конфликт product ↔ compliance не разрешим |
| F16 | **Неизменяемый audit trail** | Юридическое требование 115-ФЗ. Марина: «если ЦБ спросит — покажу скрин из Slack? Несерьёзно» |
| F26 | **RBAC: роли и доступ** | Без ролей невозможно разделить CO / CEO / Аудитор / Dev. Фундамент архитектуры |
| F32 | **REST API + Webhooks** | JTBD Дмитрия: «интегрировать за 2 дня». Без API платформа — изолированная CRM |
| F46 | **Push-нотификации** | Без уведомлений CEO не узнает об эскалации, CO не узнает об обновлении перечня. Марина: «обновление в пятницу вечером» |

**Итого Must have: 12 фич**
