## Персона 1: Марина — Compliance Officer финтех-стартапа

### 1. Роль и контекст

**Кто:** Единственный compliance-специалист в платёжном стартапе на 25 человек (seed-стадия, 3000–5000 клиентов). Подчиняется напрямую CEO. Отвечает за соблюдение 115-ФЗ, формирование ПВК, взаимодействие с Росфинмониторингом.

**Место продукта в дне:** Основной рабочий инструмент. Марина проводит в платформе 4–6 часов ежедневно: обрабатывает алерты, проверяет новых клиентов, готовит отчёты.

**Другие инструменты:** Dadata API (проверка паспортов/ИНН — доступ через Postman), Excel (ведение реестра подозрительных операций), email (коммуникация с ЦБ и коллегами), Личный кабинет Росфинмониторинга (ручная отправка ФЭС).

### 2. Job-to-be-Done

**JTBD:** «Когда мне нужно обработать 50-100 алертов в день и подготовить компанию к проверке ЦБ, я хочу инструмент, который собирает все проверки в одном месте и ведёт журнал решений, чтобы я не потеряла лицензию и не выгорела от рутины.»

**Ситуация:** Утро понедельника. Пришло обновление перечня Росфинмониторинга. 462-ФЗ даёт 24 часа на реагирование. Марина должна: пробить обновлённый список по клиентской базе → найти совпадения → заблокировать → сформировать ФЭС 2-ФМ → отправить в РФМ. Сейчас это 4 разных инструмента и 3 часа ручной работы.

**Результат успеха:** Все клиенты проверены, блокировки проведены, отчёт отправлен — за 30 минут, с полным audit trail.

### 3. Паттерны поведения

**Частота:** Ежедневно, 5 дней в неделю. Пиковая нагрузка — начало рабочего дня (новые алерты за ночь) и дни обновления перечней РФМ.

**Типичный сценарий:**
1. Открывает alert queue → фильтрует по severity (High → Medium)
2. Берёт кейс → читает карточку клиента (ФИО, ИНН, история транзакций)
3. Принимает решение: Approve / Reject / Escalate к CEO
4. При эскалации — пишет обоснование, ждёт ответ CEO
5. В конце недели — формирует ФЭС 1-ФМ по подозрительным операциям
6. Раз в квартал — готовит документы для проверки ЦБ

**Текущие инструменты:** Excel + Dadata + email + ЛК РФМ. Нет единой системы. Audit trail ведётся в Google Docs (копипаст).

### 4. Раздражители и ограничения

| Раздражитель | Источник данных |
|---|---|
| **False positive на 90%** — каждый день вручную отклоняет сотни ложных совпадений по «Иванов», «Мухаммадов» | deep_research: IDF-взвешивание, Reddit r/AMLCompliance |
| **4 инструмента для одной задачи** — Dadata отдельно, Excel отдельно, РФМ отдельно, email отдельно | brief.md: описание текущего процесса |
| **Нет Undo при ошибке** — одобрила клиента, а потом нашла новую информацию. Отменить нельзя | audit: эвристика #3 (свобода и контроль) |
| **ФЭС вручную** — XML-файлы формируются руками, ГОСТ-подпись через КриптоПро — каждый раз мучение | deep_research: SOAP vs REST, ГОСТ-криптография |
| **Бюджет на инструменты < 50K руб./мес** — Dadata «коробка» за 2.49 млн/год недоступна | deep_research: прайсинг Dadata |

**Ограничения:** Одна на весь compliance. Нет инженера для настройки API. Нет бюджета на Enterprise-решения ($160K/год Unit21 — невозможно).

**Страх при переходе:** «Если перенесу всё в новую систему и она упадёт в день проверки ЦБ — штраф до 1 млн руб. и моя дисквалификация на 3 года.»

### 5. Триггер выбора

| Этап | Что произойдёт |
|---|---|
| **Как найдёт** | Поиск: «автоматизация 115-ФЗ для стартапов». Тг-каналы по комплаенсу. Рекомендация от коллеги |
| **Что заставит попробовать** | Бесплатный tier (до 100 проверок/мес) + русский интерфейс + поддержка ФЭС |
| **Что заставит остаться** | Audit trail, который проходит проверку ЦБ. Экономия 2+ часов в день |
| **Dealbreaker** | Нет поддержки перечня РФМ. Нет ГОСТ-подписи. Цена > 50K руб./мес |

### 6. Инсайты → Дизайн-решения

1. **Потому что** Марина обрабатывает 50-100 алертов/день одна → **нужны:** keyboard shortcuts (A/R/E/N), bulk-действия, IDF-взвешивание кириллицы (снижение false positive с 90% до <15%)
2. **Потому что** она формирует ФЭС в 4 инструментах → **нужен:** BFF-прокси: JSON-запрос → XML + ГОСТ → РФМ, кнопкой из интерфейса
3. **Потому что** она боится проверки ЦБ → **нужен:** Audit Readiness Dashboard с чеклистом: «% готовности к проверке»
4. **Потому что** бюджет < 50K руб./мес → **нужен:** бесплатный tier + прозрачный прайсинг «всё включено» (без скрытых +$0.35 как у Sumsub)
5. **Потому что** у неё нет инженера → **нужен:** no-code rule builder с бизнес-языком (не «Branch 1», а «Если клиент отклонён → ...»)

## Персона 2: Алексей — CEO / основатель финтех-стартапа

### 1. Роль и контекст

**Кто:** Технический основатель BNPL-сервиса (15 человек, pre-Series A). Недавно попал в реестр операторов рассрочки ЦБ (283-ФЗ). Отвечает за всё: продукт, стратегию, фандрейзинг. Compliance — головная боль, которую хочет делегировать, но не может.

**Место продукта в дне:** Заходит 1-2 раза в неделю на 15 минут. Смотрит dashboard: «Всё ли в порядке? Есть ли эскалации, требующие моего решения? Готовы ли мы к аудиту?»

**Другие инструменты:** Notion (стратегия), Slack (коммуникация), GitHub (код), Amplitude (аналитика), Stripe/ЮKassa (платежи).

### 2. Job-to-be-Done

**JTBD:** «Когда мой CO блокирует запуск фичи из-за compliance-риска, а инвестор давит на скорость, я хочу инструмент, который покажет мне реальный уровень риска и позволит принять осознанное решение (одобрить или отклонить) с юридической страховкой, чтобы я не потерял ни лицензию, ни раунд.»

**Ситуация:** Продуктовик Дима хочет запустить упрощённый KYC (только email + телефон) для повышения конверсии. CO Марина блокирует: «Нарушение 115-ФЗ». Алексей получает эскалацию. Ему нужно: понять риск, принять решение, зафиксировать его юридически чисто.

**Результат успеха:** Решение принято за 5 минут, с пометкой «риск принят», PDF-отчёт для аудитора сгенерирован автоматически. Если ЦБ спросит — есть документ.

### 3. Паттерны поведения

**Частота:** 1-2 раза в неделю. Внеплановый вход — при эскалации (push-уведомление в Slack/email).

**Типичный сценарий:**
1. Получает уведомление: «CO отклонил запрос продуктовика. Требуется ваше решение»
2. Открывает карточку эскалации → видит diff «До/После» (текущее правило vs. предложенное изменение)
3. Читает risk assessment: «Штраф до 500K руб., вероятность проверки — средняя»
4. Принимает решение: Override (с пометкой «риск принят») или поддерживает блокировку
5. Раз в неделю — проверяет CEO Dashboard: общий risk score, тренды, прогресс audit readiness

**Текущие инструменты:** Slack-тред с Мариной. Решения «в переписке» — никакого audit trail.

### 4. Раздражители и ограничения

| Раздражитель | Источник данных |
|---|---|
| **Нет CEO Dashboard** — ни один конкурент не показывает картину для CEO | competitive_analysis_final: feature matrix |
| **Compliance = чёрная коробка** — не понимает, что делает Марина и почему блокирует | brief.md: конфликт product ↔ compliance |
| **Решения в Slack** — при аудите ЦБ нечего показать, кроме переписки | deep_research: audit trail, Four Eyes Principle |
| **Скрытый pricing конкурентов** — невозможно спрогнозировать бюджет | audit: pricing, скрытые микротранзакции Sumsub |

**Ограничения:** Времени на compliance — максимум 30 мин/неделю. Не разбирается в 115-ФЗ (для этого есть Марина). Бюджет: готов платить 30-70K руб./мес, если покажут ROI.

**Страх при переходе:** «Если compliance-инструмент сломается — это не баг, это штраф и потеря лицензии. Я не могу себе позволить эксперименты.»

### 5. Триггер выбора

| Этап | Что произойдёт |
|---|---|
| **Как найдёт** | Марина (CO) принесёт. Или увидит на конференции / в TG-канале для CEO финтехов |
| **Что заставит попробовать** | Бесплатный tier + обещание «30 минут вместо 3 часов на compliance в неделю» |
| **Что заставит остаться** | CEO Dashboard с % готовности к аудиту. Override-механизм с PDF. Ощущение контроля |
| **Dealbreaker** | Нет dashboard (нужно лезть в детали). Нет интеграции со Slack (уведомления). Цена непредсказуема |

### 6. Инсайты → Дизайн-решения

1. **Потому что** Алексей заходит на 15 мин раз в неделю → **нужен:** CEO Dashboard с 3 метриками: Risk Score, Pending Escalations, Audit Readiness %
2. **Потому что** он принимает решение по эскалации без контекста → **нужен:** diff-экран «До/После» + risk assessment с оценкой штрафа в рублях
3. **Потому что** решения сейчас в Slack → **нужен:** Override с обязательным полем «обоснование» + авто-генерация PDF для аудитора
4. **Потому что** он не эксперт в 115-ФЗ → **нужны:** human-readable объяснения: «Этот клиент в перечне террористов РФМ. Блокировка обязательна по ст. 7.5 115-ФЗ»

## Персона 3: Дмитрий — Backend-разработчик стартапа

### 1. Роль и контекст

**Кто:** Senior backend-разработчик в платёжном стартапе. Отвечает за интеграцию внешних сервисов (платежи, KYC, скоринг). Compliance — не его зона ответственности, но он единственный, кто может подключить API.

**Место продукта в дне:** Работает с продуктом 2-3 дня при подключении, потом — только при инцидентах или обновлениях API. Ожидание: настроил, забыл, работает.

**Другие инструменты:** VS Code, Postman, GitHub, Docker, PostgreSQL, ЮKassa API, Dadata API.

### 2. Job-to-be-Done

**JTBD:** «Когда CEO говорит "подключи KYC-проверку к нашему онбордингу", я хочу API с нормальной документацией и SDK, чтобы интегрировать за 1-2 дня, а не месяц, и не трогать это больше.»

**Ситуация:** Стартап получил лицензию. CEO говорит: «Нужен KYC на онбординге — до конца спринта». Дмитрий ищет API: POST /api/verify → отправить паспорт + селфи → получить ответ (approved/rejected/pending). Всё. Никаких деталей про 115-ФЗ — это задача Марины.

**Результат успеха:** Интеграция работает за 2 дня. Webhook приходит в Slack при reject. Логи в Grafana. Можно забыть.

### 3. Паттерны поведения

**Частота:** 2-3 дня интенсивной работы при подключении. Потом — раз в квартал при обновлении API или инцидентах.

**Типичный сценарий:**
1. Читает API-документацию (Swagger) → ищет эндпоинты KYC, webhook-формат, rate limits
2. Получает API-ключ → делает тестовый запрос в sandbox
3. Интегрирует: POST /verify → обрабатывает webhook → записывает статус в БД
4. Настраивает мониторинг: алерт в Slack если API отвечает >3 сек или 5xx
5. Деплоит, передаёт Марине доступ к dashboard. Забывает.

**Текущие инструменты:** Dadata API (но только проверка паспортов, без KYC-flow). Для полной верификации — самописное решение на Python + OpenCV (liveness). Месяц работы.

### 4. Раздражители и ограничения

| Раздражитель | Источник данных |
|---|---|
| **Sumsub: документация с пробелами** — стартапы жалуются на крутую кривую обучения | deep_research: отзывы на Sumsub, G2/Capterra |
| **SOAP-протокол РФМ** — интеграция с Росфинмониторингом требует месяцев, несовместима с REST-стеком | deep_research: SOAP vs REST, ГОСТ-криптография |
| **Нет sandbox** у многих конкурентов — нельзя протестировать без оплаты | audit: эвристика #10 (помощь и документация) |
| **Скрытый rate limiting** — API ломается под нагрузкой без warning | competitive_analysis: отзывы G2 |

**Ограничения:** На compliance-задачи выделено 2-3 дня в спринте. Не разбирается в 115-ФЗ (и не хочет). Если API кривой — просто возьмёт другой.

**Страх при переходе:** «Если API сломается в production в пятницу вечером — я тот, кого разбудят.»

### 5. Триггер выбора

| Этап | Что произойдёт |
|---|---|
| **Как найдёт** | Google: «KYC API Russia», Habr, GitHub awesome-lists, рекомендация в Telegram dev-чате |
| **Что заставит попробовать** | curl-пример в README. Sandbox без регистрации. Первые 100 запросов бесплатно |
| **Что заставит остаться** | Latency <300ms. Uptime 99.9%. Webhook при статусе. SDK на Python/Go |
| **Dealbreaker** | Нет Swagger. Нет sandbox. Ответ API >2 сек. Документация только на русском (без примеров кода) |

### 6. Инсайты → Дизайн-решения

1. **Потому что** Дмитрий интегрирует за 2-3 дня → **нужны:** quickstart guide «от 0 до первой верификации за 15 минут» + curl-примеры + SDK (Python, Go, Node)
2. **Потому что** он не хочет знать про 115-ФЗ → **нужна:** чёткая граница: API отвечает approved/rejected/escalated, а бизнес-логику настраивает CO в dashboard
3. **Потому что** SOAP РФМ — его кошмар → **нужен:** BFF-прокси скрыт от разработчика: он отправляет JSON, платформа сама упаковывает в XML + ГОСТ
4. **Потому что** если API ляжет — разбудят его → **нужны:** status page, <300ms latency, webhook retry с exponential backoff, rate limit headers
