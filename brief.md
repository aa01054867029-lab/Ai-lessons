# Бриф: Compliance-платформа для финтех-стартапов

> **Дата:** 2026-04-18
> **Источник:** Рекурсивный брифинг (3 раунда)
> **Юрисдикция:** 🇷🇺 РФ, ЦБ РФ, 115-ФЗ

---

## 1. Контекст продукта

Платформа для финтех-стартапов, которая упаковывает регуляторный compliance (KYC, AML, санкции, отчётность) в понятный рабочий процесс. Ключевая UX-задача — разрешить встроенный конфликт между двумя полюсами:

| Полюс | Приоритет | Риск без баланса |
|---|---|---|
| **Продукт (скорость)** | Быстрый запуск, упрощённые проверки, рост конверсии | Нарушение 115-ФЗ, штрафы ЦБ, отзыв лицензии |
| **Комплаенс (безопасность)** | Полная верификация, строгие правила, аудит-трейл | Медленный onboarding, потеря клиентов, паралич продукта |

Платформа должна предоставить **механизм компромисса** — approval workflow с эскалацией, где каждое решение прозрачно и логируется.

---

## 2. Целевые пользователи

| Роль | Задачи | Частота | Уровень доступа |
|---|---|---|---|
| **Compliance Officer** | Настройка правил, разбор алертов (50-100/день), утверждение/отклонение запросов, работа с отчётами | Ежедневно, основной пользователь | Полный доступ |
| **CEO / Founder** | Dashboard с рисками, override решений CO (с пометкой «риск принят»), стратегический контроль | Несколько раз в неделю | Управление + эскалация |
| **Внешний аудитор** | Просмотр документов, политик, логов изменений | Периодически (при проверках) | Read-only |
| **Разработчик стартапа** | Интеграция API проверок в свой продукт | При интеграции + поддержка | API + документация |

## 3. Ключевые сценарии

### 3.1 KYC/KYB-верификация
- **Модель**: агрегатор/гибрид — внешние API для проверки документов + собственные бизнес-правила (scoring, пороги)
- **Для РФ (115-ФЗ)**: проверка паспорта РФ, СНИЛС, скрининг по перечню Росфинмониторинга
- **Типовая проблема**: продуктовик хочет KYC «только email + телефон», комплаенс требует полную верификацию → решается через approval workflow

### 3.2 AML-мониторинг транзакций
- **Режим**: гибрид — realtime-алерты для сумм выше порога + daily batch для выявления паттернов (дробление, smurfing)
- **Разбор алертов**: вручную, Compliance Officer
- **Объём**: 50-100 алертов/день → требуется queue-система
- **UX-требования к queue**:
  - Фильтры (по типу, сумме, risk score, дате)
  - Приоритизация (критичные наверху)
  - Bulk-действия (массовое закрытие false positive)
  - Детальная карточка алерта с контекстом

### 3.3 Санкционный скрининг
- Автоматическая проверка клиентов по перечню Росфинмониторинга
- Алерты при совпадении → в общую queue CO

### 3.4 Regulatory reporting
- Автоматическая генерация отчётов для ЦБ РФ
- Формы в соответствии с требованиями 115-ФЗ

### 3.5 Policy management
- Ведение внутренних политик и процедур
- Версионирование документов
- Read-only доступ для аудиторов

### 3.6 Approval workflow (ключевой сценарий)
```
Продуктовик создаёт запрос на изменение правил
        │
        ▼
Compliance Officer рассматривает
        │
   ┌────┴────┐
   ▼         ▼
Утвердить  Отклонить
   │         │
   │         ▼
   │    CEO может override
   │    (с пометкой «риск принят»)
   │         │
   ▼         ▼
Правило применено
        │
        ▼
Всё логируется для аудита
(кто, когда, почему, какой риск)
```

## 4. Технические ограничения

> ⚠️ Частично не подтверждено — см. раздел «Открытые вопросы»

**Предполагаемая архитектура** (на основе типа продукта):
- Веб-приложение — dashboard для CO, CEO, аудитора
- REST API + webhooks — для разработчиков стартапов
- Документация API (Swagger / OpenAPI)
- Интеграции с внешними KYC/AML-провайдерами

## 5. UI-требования

### Дизайн-система
- **Полная свобода** — нет существующего бренд-гайда или UI-kit
- Рекомендуемый вектор: профессиональный финтех-стиль (ориентиры — Stripe Dashboard, Mercury, Unit21)

### Ключевые UX-паттерны
| Компонент | Требование |
|---|---|
| **Alert queue** | Фильтры, приоритизация, bulk-действия, детальные карточки. Основной рабочий инструмент CO |
| **Approval workflow** | Прозрачный статус запроса, эскалация к CEO, audit trail |
| **Risk dashboard (CEO)** | Высокоуровневая картина рисков, без необходимости глубокого погружения |
| **Audit view** | Read-only, логи изменений, экспорт документов |
| **API-документация** | Developer portal, примеры кода, песочница |

### Конфликт скорость/безопасность в UI
- Визуально показывать **уровень риска** каждого решения
- Если CEO делает override — **явная индикация** «риск принят» (цвет, иконка, баннер)
- Audit trail доступен в один клик из любой точки workflow

## 6. Метрики успеха (OKR)

| Метрика | Описание |
|---|---|
| **Время верификации KYC** | Среднее время от заявки до завершения проверки |
| **Время разбора алерта** | Среднее время от появления алерта до решения CO |
| **% false positive** | Доля ложных срабатываний AML-мониторинга |
| **Audit readiness** | % выполненных требований к проверке ЦБ |
| **Время согласования** | Среднее время approval workflow (запрос → решение) |
| **Количество override'ов CEO** | Индикатор конфликтов между продуктом и комплаенсом |

## 8. Открытые вопросы

> Эти вопросы не были закрыты в ходе брифинга. Требуется уточнение у PM/заказчика.

| # | Вопрос | Влияние на проектирование |
|---|---|---|
| 1 | **Бизнес-модель**: SaaS на рынок или внутренний продукт? | Multi-tenancy, биллинг, onboarding, self-service регистрация |
| 2 | **Тех. стек**: подтвердить веб + API. Нужно ли мобильное приложение? | Адаптивный дизайн vs нативное приложение |
| 3 | **CEO Dashboard**: что именно CEO хочет видеть? Risk heatmap, метрики, алерты, PDF-отчёт? | Дизайн главного экрана CEO |
| 4 | **Внешние интеграции**: какие KYC/AML-провайдеры в приоритете? (Sumsub, ЕБС, собственные?) | Архитектура интеграционного слоя |
| 5 | **Масштаб MVP**: все модули сразу или поэтапный запуск? (напр. сначала KYC → потом AML → потом reporting) | Приоритизация дизайна и разработки |
| 6 | **Онбординг стартапов**: self-service или white-glove (ручная настройка командой)? | Дизайн flow регистрации и настройки |
