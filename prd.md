## 1. Executive Summary

**ComplyFlow** — compliance-платформа для финтех-стартапов РФ, которая объединяет KYC-верификацию, AML-мониторинг по 115-ФЗ и approval workflow в одном интерфейсе.

**Проблема:** Compliance Officer финтех-стартапа работает в 8 инструментах без интеграции (Dadata, Контур.Фокус, Excel, Slack, ЛК Росфинмониторинга, КриптоПро, Email, Notion). 85-90% алертов — ложные срабатывания. Решения по спорным вопросам принимаются в Slack без audit trail. Подготовка к проверке ЦБ занимает 4 дня и бессонную ночь.

**Ключевой инсайт:** Ни один из 5 проанализированных конкурентов (Sumsub, ComplyAdvantage, Unit21, Salv, Ondato) не имеет approval workflow для разрешения конфликта между продуктовой командой и комплаенсом. Все платформы рассчитаны на одного пользователя (CO), без поддержки командного принятия решений.

**Решение:** MVP из 12 фич: KYC-верификация + перечень РФМ + alert queue с bulk-действиями + Maker-Checker approval workflow с эскалацией к CEO + неизменяемый audit trail + REST API + push-нотификации.

**Primary-персона:** Марина, единственный CO в стартапе на 25 человек.

## 5. MVP Scope

**MVP в одном предложении:** ComplyFlow v1.0 позволяет Compliance Officer проверять клиентов по перечню РФМ, обрабатывать AML-алерты в единой очереди и согласовывать спорные решения через Maker-Checker workflow с эскалацией к CEO — с полным audit trail.

### Must Have фичи

| # | Фича | Описание |
|---|---|---|
| F01 | KYC-верификация | Паспорт РФ → OCR → проверка подлинности |
| F02 | Перечень РФМ | Скрининг по перечню экстремистов/террористов |
| F04 | Проверка по МВД | Действительность паспорта |
| F07 | Мониторинг транзакций | Rule-based алерты (>600K, частота, аномалии) |
| F08 | Alert queue | Очередь с risk scoring, фильтры, severity |
| F09 | Bulk-действия | Массовое одобрение/отклонение |
| F13 | Approval workflow | Запрос → CO рассматривает → Approve/Reject |
| F15 | Эскалация + override | CO Reject → CEO Override с «риск принят» |
| F16 | Audit trail | Неизменяемый журнал решений |
| F26 | RBAC | CO, CEO, Аудитор (read-only), Dev (API) |
| F32 | REST API + Webhooks | POST /verify, GET /alerts, Webhooks |
| F46 | Push-нотификации | Email + Slack: эскалации, алерты |

### Основной сценарий (Марина)

1. **Утро** → Марина открывает Dashboard CO → видит: 14 новых клиентов, 42 алерта (3 high), 1 запрос на согласование
2. **Alert queue** → фильтрует по severity → берёт 🔴 high → открывает карточку кейса
3. **Карточка кейса** → видит данные клиента + совпадение с перечнем + расхождение по дате рождения → Approve (ложное)
4. **Bulk** → выделяет 15 🟢 low-risk алертов → массовый Approve
5. **Approval request** → уведомление: «Дима-продукт запрашивает изменение правила KYC» → Reject с обоснованием «ст. 7 115-ФЗ»
6. **Эскалация** → CEO Алексей получает уведомление → видит diff + отказ CO + обоснование → решает: поддержать блокировку
7. **Audit trail** → все решения зафиксированы с датами, ролями, обоснованиями

### Что осознанно НЕ входит

| Фича | Версия | Почему не сейчас |
|---|---|---|
| Генератор ФЭС (F19) | v1.1 | 2-3 мес инженерной работы (XML + ГОСТ + SOAP). CO пока формирует руками |
| IDF-взвешивание (F10) | v1.1 | ML-задача, требует корпуса русских имён. Bulk-действия компенсируют |
| CEO Dashboard (F24) | v1.1 | CEO получает уведомления + override. Dashboard — удобство |

## 6. User Stories

### F01 — KYC-верификация

**US-01:** Как Compliance Officer, я хочу загрузить фото паспорта клиента и получить автоматическую проверку подлинности, чтобы не проверять каждый паспорт вручную.

Acceptance Criteria:
- [ ] CO загружает фото паспорта (JPEG/PNG) через интерфейс
- [ ] Система извлекает: ФИО, дата рождения, серия/номер, дата выдачи
- [ ] Система проверяет паспорт по базе МВД (действительность)
- [ ] Результат отображается на карточке клиента: ✅ Валиден / ❌ Недействителен / ⚠️ Требует проверки
- [ ] Время обработки: < 10 сек

### F02 — Перечень РФМ

**US-02:** Как Compliance Officer, я хочу, чтобы каждый клиент автоматически проверялся по перечню Росфинмониторинга, чтобы я не пропустила совпадение и не получила штраф.

Acceptance Criteria:
- [ ] При создании клиента — автоматический скрининг по перечню
- [ ] Совпадение по ФИО → алерт в alert queue с severity High
- [ ] На карточке совпадения: данные клиента vs. данные из перечня (ФИО, дата рождения, ИНН)
- [ ] CO может отметить: «Ложное срабатывание» или «Подтверждённое совпадение → блокировка»
- [ ] Перечень обновляется ежедневно (автозагрузка с сайта РФМ)

### F07 — Мониторинг транзакций

**US-03:** Как Compliance Officer, я хочу получать алерты при подозрительных транзакциях, чтобы выявлять потенциальные нарушения 115-ФЗ.

Acceptance Criteria:
- [ ] Предустановленные правила: сумма >600K, >5 переводов/день, новый клиент + крупная операция
- [ ] Алерт создаётся автоматически при срабатывании правила
- [ ] Алерт содержит: клиент, правило, сумма, время, история транзакций за 30 дней
- [ ] Risk score: автоматический расчёт на основе кол-ва сработавших правил

### F08 — Alert queue

**US-04:** Как Compliance Officer, я хочу видеть все алерты в единой очереди с фильтрами и приоритизацией, чтобы обрабатывать сначала критичные.

Acceptance Criteria:
- [ ] Единая очередь всех алертов (KYC + транзакции + перечень)
- [ ] Фильтры: по severity (High/Medium/Low), по типу, по дате, по статусу
- [ ] Сортировка: по severity (по умолчанию), по дате, по клиенту
- [ ] Счётчик: «42 алерта (3 high, 12 medium, 27 low)»
- [ ] Клик на алерт → открывается карточка кейса

### F09 — Bulk-действия

**US-05:** Как Compliance Officer, я хочу массово одобрять/отклонять однотипные алерты, чтобы не тратить 3 минуты на каждый ложный.

Acceptance Criteria:
- [ ] Чекбоксы у каждого алерта в очереди
- [ ] «Выбрать все» с фильтром (например, все Low severity)
- [ ] Кнопки: «Approve выбранные», «Reject выбранные»
- [ ] Обязательный комментарий при bulk reject (одно поле на все)
- [ ] Каждое действие фиксируется в audit trail (даже bulk)

### F13 — Approval workflow

**US-06:** Как Product Manager, я хочу запросить изменение compliance-правила через формализованный процесс, чтобы не блокироваться на переписке в Slack.

Acceptance Criteria:
- [ ] PM создаёт запрос: «Изменить правило X → на Y» + обоснование
- [ ] Запрос приходит CO как уведомление
- [ ] CO видит: кто запросил, что именно меняется, обоснование
- [ ] CO выбирает: Approve (с комментарием) или Reject (с обоснованием)
- [ ] Результат фиксируется в audit trail

### F15 — Эскалация + Override

**US-07:** Как CEO, я хочу видеть эскалированные запросы и принимать финальное решение, чтобы разрешить конфликт между продуктом и комплаенсом.

Acceptance Criteria:
- [ ] При Reject CO → PM может нажать «Эскалировать к CEO»
- [ ] CEO получает уведомление с: запрос PM + отказ CO + обоснование CO
- [ ] CEO выбирает: «Поддержать блокировку» или «Override — принять риск»
- [ ] Override требует обязательное текстовое обоснование
- [ ] Решение CEO — финальное, фиксируется в audit trail с меткой «CEO override»

### F16 — Audit trail

**US-08:** Как Compliance Officer, я хочу, чтобы все решения автоматически фиксировались в неизменяемом журнале, чтобы я могла показать его аудитору ЦБ.

Acceptance Criteria:
- [ ] Каждое действие записывается: дата, пользователь, роль, действие, объект, обоснование
- [ ] Записи нельзя удалить или изменить (immutable log)
- [ ] Фильтрация журнала: по дате, по пользователю, по типу действия
- [ ] Экспорт за период: JSON или CSV
- [ ] Retention: минимум 5 лет (требование 115-ФЗ)

### F26 — RBAC

**US-09:** Как администратор, я хочу назначать роли пользователям, чтобы каждый видел только то, что ему нужно.

Acceptance Criteria:
- [ ] 4 роли: CO (полный доступ), CEO (dashboard + override + read), Аудитор (read-only), Dev (только API)
- [ ] CO может приглашать пользователей и назначать роли
- [ ] Аудитор видит: audit trail, журнал решений. Не может менять данные
- [ ] Dev получает API-ключ, без доступа к dashboard

### F32 — REST API + Webhooks

**US-10:** Как Backend-разработчик, я хочу интегрировать верификацию через REST API, чтобы подключить KYC к нашему онбордингу за 2 дня.

Acceptance Criteria:
- [ ] POST /api/v1/verify — отправить данные клиента + фото документа
- [ ] GET /api/v1/clients/{id}/status — получить статус верификации
- [ ] GET /api/v1/alerts — получить список алертов (с пагинацией)
- [ ] Webhook: POST на URL клиента при изменении статуса (approved/rejected/pending)
- [ ] API-ключ для аутентификации
- [ ] Rate limit: 100 req/sec, headers X-RateLimit-*
- [ ] Latency: < 300ms (p95)

### F46 — Push-нотификации

**US-11:** Как CEO, я хочу получать уведомления при эскалациях, чтобы не пропустить запрос, требующий моего решения.

Acceptance Criteria:
- [ ] Каналы: Email + Slack (настраивается в профиле)
- [ ] Триггеры: эскалация к CEO, обновление перечня РФМ, High-severity алерт
- [ ] CO настраивает свои триггеры: новый алерт High, обновление перечня, approval request
- [ ] Ссылка в уведомлении ведёт на конкретный кейс/запрос
- [ ] Возможность mute на время (Do Not Disturb)

## 7. Нефункциональные требования

### Производительность
- **API latency:** < 300ms (p95), < 1s (p99)
- **Dashboard load:** < 2 сек
- **Alert queue:** рендер 100 алертов < 1 сек
- **Concurrent users:** ≥ 50

### Безопасность
- **Шифрование:** TLS 1.3 (transit), AES-256 (at rest)
- **Аутентификация:** Email + пароль + TOTP (2FA обязательно для CO/CEO)
- **Audit trail:** Immutable, append-only (запись в отдельную БД/сервис)
- **Персональные данные:** хранение по 152-ФЗ, обезличивание в логах
- **Retention:** ≥ 5 лет для журнала решений (требование 115-ФЗ)
- **ГОСТ-совместимость:** подготовка к интеграции с КриптоПро (v1.1)

### Доступность
- **SLA:** 99.9% uptime (не более 8.7 часов даунтайма/год)
- **Status page:** публичная страница статуса сервиса
- **Backup:** ежедневный, retention 30 дней

### Платформы
- **Web:** Chrome, Firefox, Safari, Edge (последние 2 версии)
- **Адаптивность:** Desktop-first (CO работает на ноутбуке). Мобильная версия — read-only для CEO
- **API:** REST, JSON, OpenAPI 3.0

## 8. Риски и открытые вопросы

### Риски

**🔴 Высокий:**

- **Регуляторное изменение 115-ФЗ** — новые требования могут потребовать срочной переработки. Митигация: подписка на обновления ЦБ, модульная архитектура правил.
- **Качество OCR для российских паспортов** — варианты: собственная модель (дорого) vs. сторонний API (Dadata, Smart IDReader). Нужно протестировать accuracy на 100+ реальных документах.
- **CO не покупает сам — CEO покупает** — маркетинг должен убеждать CO, который убеждает CEO. Нужен ROI-калькулятор для CEO.

**🟡 Средний:**

- **Юридическая ответственность за решения** — если платформа авто-одобрила клиента, а он оказался в перечне → кто отвечает? Нужно Terms of Service + disclaimers.
- **Glобальные конкуренты зайдут в РФ** — Sumsub может добавить 115-ФЗ. Митигация: speed to market + depth of 115-ФЗ интеграции.
- **False positive при IDF-взвешивании** — если алгоритм пропустит реальное совпадение → штраф. Нужен human-in-the-loop для Medium/High cases.

### Открытые вопросы (требуют кастдева)

| # | Вопрос | Как узнать |
|---|---|---|
| 1 | Keyboard shortcuts — реально ли нужны CO, или переключение инструментов — бóльшая боль? | Наблюдение за CO в реальном окружении (contextual inquiry) |
| 2 | Какой формат audit trail ожидает ЦБ при проверке? | Интервью с ex-инспектором ЦБ или compliance-консультантом |
| 3 | Готовы ли CO платить за облачную ГОСТ-подпись или привязаны к локальному КриптоПро? | Опрос 10-15 CO (квант-исследование) |
| 4 | ПДЛ-база: собирать свою или интегрировать Контур.Фокус? | Анализ покрытия и стоимости Контур.Фокус API |
| 5 | Compliance-обучение сотрудников: нужен LMS или достаточно треккера? | Интервью с 3-5 CO (каков текущий процесс обучения?) |
