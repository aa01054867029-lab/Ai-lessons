# Директива: Сборка экрана из базы (screens)

> **Запуск:** только когда пользователь явно дал эту директиву и ссылку на экран. Переиспользуемая: запускается на каждый новый экран. Источник — дизайн экрана в Figma + база `src/components/` + правила `ds/CONTRACT.md`.

## Задача
Собрать экран продукта в React **из компонентной базы** по его дизайну в Figma и подключить к проекту. Агент читает дизайн, узнаёт в нём компоненты базы, выкладывает страницу по раскладке макета, регистрирует её рядом с остальными экранами **и выводит на навигационную страницу-индекс, с которой видно и открывается каждый собранный экран** (плиточная навигация, не россыпь файлов в папке). Результат — статичный, но настоящий экран: как выглядит и как держится по ширине и контенту. Поведение, навигация и данные — не здесь (директива оживления).

## Когда применять
- После `react_base`: есть база `src/components/` и токены, проект поднимается. Каталог (`storybook`) для сборки экранов не обязателен, но удобен для сверки.
- На **каждый** экран — отдельный запуск с ссылкой на этот экран. Повторный запуск на уже собранный экран — обновляет его (Фаза 7), на новый — добавляет и подключает к остальным.
- Это **стоячий инструмент, не разовый прогон.** Возвращаешься к нему весь проект: собрал один экран, потом второй, потом пересобрал изменившийся. Реестр копит все экраны; состояние не затирается. (Позже накопленные директивы свернутся в skills — но логика та же: переиспользуемый ход.)

## Предусловие
- Собрана база (`directive_react_base`).
- Figma MCP подключён (`directive_figma_mcp`) — дизайн читается через него.
- Дизайн экрана существует в Figma как отдельный фрейм.

## Ключевой принцип
Экран — **композиция из базы**, не рисунок с нуля. Что есть в базе — берётся инстансом/компонентом; чего нет (доменные блоки вроде карточки сервера) — собирается как локальная композиция из той же базы и токенов. Раскладка переносится из автолейаутов и констрейнтов макета в отзывчивую вёрстку (FILL/HUG/FIXED → flex grow/hug/fixed, wrap → flex-wrap), а не в прибитые координаты. Качество кода = чистота макета: чем аккуратнее автолейауты и компоненты в Figma, тем точнее перенос.

## Граница (важно — что НЕ делаем)
Директива собирает ТОЛЬКО раскладку и статичный вид экрана. Она **НЕ**:
- переносит/пишет логику, не вешает обработчики;
- строит слой данных и состояние, не генерирует мок-датасеты, не делает контент динамическим;
- делает рабочие формы, навигацию-как-действие, переходы по клику.
Всё это — **директива оживления (следующий урок)**, не здесь.

**Почему так:** двигаемся итеративно и по одному. Если попросить агента «собери и заодно оживи», он соберёт экраны и тут же начнёт навешивать поведение и данные — и следующий шаг останется без работы («да тут уже всё готово»). Сначала аккуратно собранные статичные экраны, потом отдельно — логика. Так задачи разграничены, и на каждом шаге фокус на одном процессе. Даже если агенту «логично сделать сразу» — на этом шаге не делает.

## Фаза 0. Контекст
1. Прочитай `ds/CONTRACT.md` и `ds/components.md`. Посмотри `src/components/index.ts` — что есть в базе.
2. Убедись, что в проекте есть каркас экранов: роутер, реестр (`src/screens/registry.ts`) и **навигационная страница-индекс** на корневом роуте `/`, которая строится из реестра. Нет — заведи минимальный (router + пустой реестр + пустой индекс). Это «среда для экранов»: каждый собранный экран подключается в реестр, а индекс-страница автоматически показывает его плиткой. Индекс — обязательная часть результата, не опция.

## Фаза 1. Вход — ссылка на экран
Один запрос:
```
Ссылка на дизайн экрана — на сам фрейм экрана (правый клик по фрейму → Copy link to selection), не на корень файла.
```
Если ссылка на корень файла — переспроси. **По корню MCP отдаёт только первую страницу** — получишь не тот контент. Нужна ссылка на конкретный фрейм экрана. Уточни имя экрана (для папки/роута), если из дизайна не ясно.

## Фаза 2. Прочитать дизайн
1. `get_metadata` по ссылке — карта структуры фрейма (секции, вложенность), без тяжёлого контента.
2. `get_design_context` по секциям (не весь фрейм разом, если он большой — фильтруй по регионам: header, body, секции). Забираешь раскладку, имена слоёв, токены, инстансы компонентов.
3. `get_screenshot` — визуальный референс, чтобы сверять результат глазами.

## Фаза 3. Декомпозиция
1. Пройди структуру экрана сверху вниз. Каждый блок сопоставь с компонентом базы **по имени/структуре** (инстанс `Button` в макете → `<Button>` из базы; `Card`, `Navbar`, `Badge`, `Table` и т.д.).
2. Блоки, которых нет в базе (доменные композиции — ServerCard, StatCard, ActivityRow и подобные): собери их как **локальные компоненты экрана** (`src/screens/<name>/parts/`) из компонентов базы + токенов. В базу их не тащи. Пометь как кандидатов на вынос (если такой блок встретится на втором экране — повод вынести в базу).
3. Не подменяй: нет точного совпадения — лучше локальная композиция, чем натянуть чужой компонент. Базу при этом не перерисовываешь.

## Фаза 4. Сборка страницы
1. Выложи страницу по раскладке макета: автолейауты → flex/grid, направление/выравнивание/gap → соответствующие свойства, FILL/HUG/FIXED → `flex: 1`/`hug`/фикс. размеры, `layoutWrap` → `flex-wrap`, констрейнты → отзывчивое поведение. Цель — экран держится при смене ширины и объёма контента, а не только в одном размере.
2. Стили — **только семантические токены** (`var(--surface-default)`, `--space-*`, `--radius-*`), как требует `CONTRACT`. Ни одного прибитого значения, где есть токен.
3. Контент — **статикой, ровно как в макете** (те же тексты и значения, что в дизайне), чтобы экран рендерился и сверялся с Figma. НЕ выдумывай дополнительные данные, НЕ строй слой данных или состояние, НЕ генерируй датасеты. Никакой логики, обработчиков, рабочих форм, навигации-как-действия, динамики — это директива оживления (см. «Граница»). Картинки/иконки — плейсхолдеры (визуал не здесь).

## Фаза 5. Подключить к проекту и вывести на индекс
1. Положи страницу в `src/screens/<name>/<Name>.tsx`.
2. Зарегистрируй экран в реестре `src/screens/registry.ts` — запись с полями: `id`, `name` (название экрана для плитки), `description` (одна строка — что на экране: ключевые блоки), `route` (путь), `component`. Реестр — единый источник и для роутинга, и для индекс-страницы.
3. Обнови **навигационную страницу-индекс** (корневой роут `/`): она строится из реестра, поэтому новый экран появляется на ней плиткой сам. Плитка = название + краткое описание + путь, вся плитка кликабельна и ведёт на экран. Индекс-страницы ещё нет — собери её (см. ниже). Это и есть «подключение к остальным» так, чтобы экраны было удобно смотреть, а не искать в папке.
4. Запиши короткую заметку экрана `src/screens/<name>/<name>.md`: какие компоненты базы использованы, какие локальные доменные блоки, какой роут, что mock. Это держит проект самодокументированным и помогает повторным запускам.

### Навигационная страница-индекс (обязательна)
Одна страница на корневом роуте `/`, собранная на компонентах базы и токенах (выглядит консистентно с продуктом, не служебная заглушка). Что на ней:
- Заголовок — название продукта (например, «CryptoVPS — экраны»).
- Подзаголовок: «Статичные экраны, собранные из дизайн-системы. Логика и данные — отдельный шаг.»
- Сетка карточек-плиток по всем экранам **из реестра**: на каждой — название, краткое описание (что на экране), путь/роут; вся карточка кликабельна и ведёт на экран.
- Внизу — ссылка на DS-витрину (Storybook/Showcase), если каталог поднят (предыдущий урок); нет — пропусти.

Страница перестраивается из реестра при каждом добавлении экрана — плитки руками не дублируй и не хардкодь. Цель: студент запустил директиву и сразу получил удобную точку входа, с которой открывает любой собранный экран.

## Фаза 6. Поднять и проверить
1. `npm run dev`, открой роут экрана.
2. Проверь сам: рядом с дизайном в Figma — компоненты те, раскладка та, отступы/сетка совпадают. Затем потяни окно по ширине и подставь контент подлиннее (заголовок, больше элементов в списке) — раскладка должна держаться. Консоль без ошибок.
3. Покажи пользователю роут и результат. Расхождения он называет словами — правишь через агента, в код он не лезет.

## Фаза 7. Повторный запуск
- **Новый экран** — повтори Фазы 1–6, добавь в реестр, остальные экраны не трогай.
- **Тот же экран** (обновился дизайн) — пересобери только его страницу и заметку, роут не дублируй. В финале перечисли, что изменилось.
Реестр — единый список экранов проекта; именно через него среда остаётся унифицированной по мере роста.

## Self-check (перед финалом)
- Каждый блок экрана собран из базы или из локальной композиции на базе? Прибитых компонентов «мимо базы» нет?
- Прогон по коду: нет прибитых hex/px, где есть токен?
- Экран зарегистрирован в реестре/роутере и открывается; остальные экраны не сломаны?
- **Навигационная индекс-страница на `/` обновлена: новый экран есть плиткой, плитки кликабельны и ведут на экраны? Индекс строится из реестра, а не захардкожен?**
- Раскладка держится на разной ширине и при контенте длиннее макетного?
- **Нет логики, обработчиков, слоя данных/состояния, рабочих форм?** Контент — статикой из макета, не из данных. Если агент начал оживлять — откати, это следующий урок.
- `npm run dev` без ошибок в консоли.

```

- Из базы: <перечень компонентов>
- Локальные доменные блоки: <перечень или «нет»> (кандидаты на вынос: <…>)
- Заметка: src/screens/<name>/<name>.md
- Индекс: открой `/` — экран на месте плиткой (название + описание + путь), переход работает
Проверка раскладки (ширина/контент): <ок / замечания>
Логику и данные не трогал — экран статичный (оживление — отдельная директива).
Не собралось / требует внимания: <список или «нет»>
```

## Грабли (учтено)
- **Ссылка на корень файла** → MCP вернёт только первую страницу, экран не прочитается. Требуем ссылку на сам фрейм (Copy link to selection).
- **Большой фрейм** → `get_design_context` на весь экран переполнит контекст. Сначала `get_metadata` (карта), потом контекст по секциям.
- **Нет компонента под блок** → локальная композиция на базе, не подмена чужим компонентом и не перерисовка базы.
- **Прибитые координаты вместо раскладки** → перенос ломается при смене ширины. Переносим автолейаут/констрейнты в отзывчивую вёрстку, проверяем растягиванием окна (это и есть проверка из урока).
- **Дубликат роута при повторе** → перед добавлением сверяйся с реестром; тот же экран обновляем, не плодим.
- **Агент норовит сразу оживить** (повесить обработчики, сделать данные динамическими, собрать рабочую форму) — это главная грабля шага: разрушает следующий урок. Держим статику; контент — из макета, без слоя данных. Всё динамическое — оживление.
- **Экраны собраны, но раскиданы по папкам без связующей страницы** → студент «куда-то их сложил» и не видит результат удобно. Навигационная индекс-страница на `/` обязательна и строится из реестра — с неё открывается каждый экран плиткой (название + описание + путь). Без явного предписания агент это делает не всегда — поэтому оно зашито здесь.

## Правила
- Экран — композиция из базы; база при сборке экранов не перерисовывается.
- Только семантические токены (`CONTRACT`); прибитых значений нет.
- Figma — read-only.
- Реестр экранов — единая точка: каждый собранный экран регистрируется в нём.
- Навигационная индекс-страница на `/` поддерживается всегда и строится из реестра — каждый собранный экран виден на ней кликабельной плиткой.
- Статика: ни логики, ни слоя данных, ни поведения, ни генерации датасетов — даже если кажется «логично сразу». Это директива оживления, следующий урок.
- Designer направляет и смотрит результат; правки в код вносит агент.
