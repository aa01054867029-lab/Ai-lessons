# Директива: Аудит и консистентность между экранами (screens_audit)

## Задача
Просканировать все собранные экраны проекта в SectionNode «Screens», сверить с каталогом `ds/components.md` и Foundation, найти расхождения по шести стандартным проверкам и вернуть структурированный отчёт. Опционально починить часть найденного — только через явный апрув в чате на каждый автофикс.

**Главное отличие от других директив курса.** Эта — read-only по умолчанию. Она не пишет в Figma и в каталог без явного «да» от пользователя на каждой правке. Сначала отчёт, потом — точечные автофиксы по выбору.

Директива работает в двух режимах:
- **Audit only** — `screens_audit` без аргументов: проходит по всем экранам в SectionNode «Screens», возвращает полный отчёт по шести проверкам, ничего не правит. Это режим по умолчанию
- **Audit + autofix** — `screens_audit --fix`: после отчёта по каждой найденной автофиксимой проблеме предлагает починить, ждёт «да/нет» в чате. На «да» — чинит, на «нет» — оставляет с пометкой

## Роль
Дизайн-системный аудитор с практикой работы со зрелыми ДС. Знает, что главный антипаттерн — «почти такие же, но чуть другие» компоненты, и ловит их в первую очередь. Различает осмысленные смысловые различия (Card/Default vs Card/Interactive) и накопленный мусор (два Card-варианта одного смысла, появившиеся в разное время). Никогда не правит молча — каждая правка через явный апрув.

> Адаптируй роль под продукт:
> - B2C веб → акцент на консистентность hero, форм, корзины; едва заметные расхождения в кнопках критичны для конверсии
> - B2B SaaS → акцент на табличные паттерны, фильтры, состояния пустых данных, состояния доступа по ролям
> - Мобильное приложение → акцент на edge-состояния (офлайн, медленный интернет, отказ в разрешениях), нативные паттерны iOS/Android
> - Внутренняя админка → меньше визуальной строгости, больше покрытия граничных случаев (длинные значения, многоколоночность, большие списки)

---

## Выбор режима

### Audit only: `screens_audit`

Используется по умолчанию.

- Контрольный прогон после нового экрана через `final_screens`
- Финальный аудит перед передачей в разработку
- Раз в спринт на больших проектах
- Хочешь сначала увидеть картину, потом решать, что чинить

### Audit + autofix: `screens_audit --fix`

Используется когда:
- Уже знаешь, что после прошлого аудита накопилось много мелких расхождений
- Хочешь пройтись и починить их за один прогон
- Готов в реальном времени отвечать «да/нет» на каждый предложенный фикс

В этом режиме директива всё равно не молчит — каждый автофикс отдельно подтверждается. Это не «авто-исправление всего», это «прогон с предложениями».

## Фаза 0. Читаем входы

### Фаза 0.1: SectionNode «Screens»

Через `get_metadata` найди в текущем Figma-файле SectionNode «Screens». Внутри — фреймы с именами `Screen/<Name>`. Сохрани их список:

```
screens = [
  { name: "Screen/Dashboard", nodeId: "1:bbb", composition: ds/screens/dashboard.md },
  { name: "Screen/Files",     nodeId: "1:ccc", composition: ds/screens/files.md },
  { name: "Screen/Settings",  nodeId: "1:ddd", composition: ds/screens/settings.md },
  { name: "Screen/Landing",   nodeId: "1:eee", composition: ds/screens/landing.md },
]
```

Если SectionNode «Screens» отсутствует — вернись с сообщением «нечего аудировать, сначала собери экраны через `final_screens`». Не падай молча.

### Фаза 0.2: Foundation

Через `get_metadata` найди SectionNode «Foundation». Прочти Component Sets (имена, Node ID), Variables (Primitive и Semantic). Сохрани:

```
foundation = {
  componentSets: { "Button": "5:001", "Card/Default": "5:002", ... },
  variables: { semantic: [...], primitive: [...] },
  effectStyles: ["shadow-sm", "shadow-md", "shadow-lg"],
}
```

### Фаза 0.3: Каталог

Прочти `ds/components.md`, `ds/foundation.md`, все файлы из `ds/screens/`. Сохрани в локальные структуры. Это «правда» по версии markdown — на следующих фазах будем сверять с Figma.

## Фаза 1. Шесть проверок

По каждой проверке агент проходит по всем экранам и собирает находки. На этой фазе ничего не пишет — только собирает.

### Проверка 1: Отвязанные элементы вместо instance

Для каждого экрана через `get_metadata` пройдись по детям. Найди узлы типа `FRAME`, `RECTANGLE`, `GROUP`, которые **визуально имитируют** существующий компонент из `foundation.componentSets` (по размерам, заливке, тексту, иконке внутри).

Алгоритм распознавания:
- Размер близок к размеру одного из мастеров (±10px по обеим осям)
- Цвет fill близок к одному из Semantic-токенов (по hex)
- Внутренняя структура (текст + иконка / две колонки текста / прямоугольник + текст) совпадает с анатомией мастера

Если совпадает по 2+ признакам — это кандидат на «отвязанный элемент». Записываем:

```
finding = {
  check: 1,
  screen: "Dashboard",
  node: "Hero/Frame/Button-imitator",
  suggestedComponent: "Button/Primary/Lg",
  confidence: high | medium | low,
}
```

### Проверка 2: Прибитый hex без Variable

Для каждого экрана пройди по всем нодам через `get_metadata`. На каждом fill, stroke, effect проверь: bound к Variable или прибитый hex?

Если прибитый — сверь значение со списком Semantic-токенов. Если совпадает (или совпадает с погрешностью ±5 единиц по RGB) — это кандидат на автофикс.

```
finding = {
  check: 2,
  screen: "Settings",
  node: "FormBackground/Rectangle",
  hexValue: "#fafafa",
  suggestedSemantic: "surface-subtle",
  confidence: high (exact match),
}
```

### Проверка 3: Дубликаты компонентов

В `foundation.componentSets` найди пары с похожими именами или анатомией. Эвристика:
- Похожие имена: «Card/Stats» и «StatCard», «Button/Primary» и «PrimaryButton»
- Похожая анатомия: оба имеют title + value + badge внутри, оба используют schiriiy Semantic-токены

Записываем как **подозрение**, не как баг — финальное решение за дизайнером:

```
finding = {
  check: 3,
  pair: ["Card/Stats", "StatCard"],
  similarity: 0.85,
  reason: "Похожие имена + одинаковая анатомия (title + value + badge)",
}
```

### Проверка 4: Неиспользуемые компоненты

Для каждого Component Set в `foundation.componentSets` пройди по всем экранам и проверь, есть ли хоть один instance, ссылающийся на него. Параллельно сверь с записями `used in` в `ds/components.md`.

```
finding = {
  check: 4,
  component: "IconButton",
  usedIn: [],  // ни на одном экране
  recordedUsedIn: "screens/dashboard, screens/files",  // в md записано, но в Figma не подтверждается
  status: "stale" | "unused",
}
```

`stale` — в md записано, в Figma нет (расхождение каталога). `unused` — нигде нет.

### Проверка 5: Расхождения в состояниях

Для каждого Component Set проверь, какие состояния (`state` / `variant`) реально используются на экранах. Сравни с матрицей в `ds/components.md`.

Например: в Component Set Button есть state default | hover | disabled | loading. На всех экранах встречается только `default`. Это сигнал «hover/disabled/loading определены в ДС, но нигде на экранах не используются — либо избыточны, либо забыли применить».

Обратный сценарий: на каком-то экране Button используется в кастомном состоянии, отвязанном от мастера. Это сигнал «либо в ДС не хватает варианта, либо случайно отвязали instance».

```
finding = {
  check: 5,
  component: "Button",
  type: "unused-variants" | "custom-detached",
  detail: "В ДС есть hover/disabled, ни один не используется на экранах",
}
```

### Проверка 6: Покрытие состояний (edge cases)

Для каждого экрана прочти `ds/screens/<name>.md` и пройди по секциям. На каждой секции определи **тип блока** по контексту (список / форма / карточка контента / авто-обновляемая зона / доступ) и сверь с матрицей покрытия:

| Тип блока | Обязательные состояния |
|-----------|----------------------|
| Список / таблица / лента | с данными, пусто, загрузка, ошибка, переполнение |
| Форма | по умолчанию, валидные поля, ошибки полей, отправка, успех, ошибка отправки |
| Карточка контента | по умолчанию, загрузка, ошибка, удалённый элемент |
| Авто-обновляемая зона | первое появление, загрузка, устаревшие, свежие |
| Доступ | по умолчанию, запрещён, заблокировано, требуется вход, требуется оплата |

Для каждой секции проверь, какие состояния заложены (через `visible:false` соседи, через отдельные варианты компонентов или через явные пометки в `ds/screens/<name>.md`). Чего нет — записывай как пропуск.

Контекстная адаптация: для маркетингового лендинга многие из этих состояний не нужны. Для admin-форм — критичны все. Аудит должен сам понимать контекст по типу экрана и не требовать «empty state» для hero-секции.

```
finding = {
  check: 6,
  screen: "Files",
  section: "FilesTable",
  blockType: "список",
  hasStates: ["с данными", "пусто"],
  missingStates: ["загрузка", "ошибка", "переполнение"],
  contextNote: null,  // не маркетинговая страница, состояния обязательны
}
```

## Раздел 6: Покрытие состояний

- Dashboard / StatsRow (авто-обновляемая зона): первое появление ✗, загрузка ✗, устаревшие ✗, свежие ✗. Добавить?

После отчёта спроси у пользователя, что делать дальше. По каждой автофиксимой находке предлагай:

- **Пропустить** — оставить как есть, добавить пометку в md
- **Не решать сейчас** — оставить в отчёте, чтобы вернуться позже

## Фаза 3. Автофиксы (только в режиме `--fix` и только с апрувом)

Для каждой найденной проблемы, по которой пользователь сказал «чинить», выполни конкретный автофикс через `use_figma`:

### Фикс для проверки 1: отвязанный → instance

```js
const targetNode = figma.getNodeById("<findingNodeId>");
const masterComponent = figma.getNodeById("<suggestedComponentId>");
const newInstance = masterComponent.createInstance();

// Сохраняем позицию и размер
newInstance.x = targetNode.x;
newInstance.y = targetNode.y;
newInstance.resize(targetNode.width, targetNode.height);

// Вставляем в parent на то же место
const parent = targetNode.parent;
const index = parent.children.indexOf(targetNode);
parent.insertChild(index, newInstance);

targetNode.remove();
```

### Фикс для проверки 2: прибитый hex → Variable

```js
const node = figma.getNodeById("<findingNodeId>");
const semanticVar = figma.variables.getLocalVariables()
  .find(v => v.name === "<suggestedSemantic>");

const fills = JSON.parse(JSON.stringify(node.fills));
fills[0] = figma.variables.setBoundVariableForPaint(fills[0], "color", semanticVar);
node.fills = fills;
```

### Фикс для проверки 4: unused-компонент

Это не Figma-операция, это запись в каталог. Обнови `ds/components.md`:

```markdown
### IconButton
- ...
- **Used in:** (нет)
- **Status:** не используется в текущих экранах, кандидат на удаление при следующем аудите
```

### Фикс для проверки 6: состояния

Не автофикс. Запускай отдельный прогон `final_screens` на затронутом экране с конкретным брифом «расширь состояния на секции <name>». Это уже работа `final_screens`, не аудита.

## Фаза 4. Запись в каталог

После всех апрувленных автофиксов:

### Фаза 4.1: Сохранение отчёта

Сохрани полный отчёт (вместе с результатами автофиксов — что починили, что оставили) в `.tmp/screens_audit_<YYYY-MM-DD>.md`. Это нужно для:
- Следующего прогона — чтобы показать «было / стало»
- Истории — что когда правили
- Документирования решений «не чинить» — чтобы при следующем аудите не задавать те же вопросы

### Фаза 4.2: Обновление `ds/components.md`

По каждому компоненту, которому мы добавили или сняли пометки:
- Поле `used in` — если автофикс добавил instance, обнови `used in`
- Поле `Status` — если пометили как «кандидат на удаление», запиши

### Фаза 4.3: Обновление `ds/screens/<name>.md`

По каждому решению пользователя «оставить состояние осознанно пропущенным» — добавь в файл экрана пометку:

```markdown
## Edge cases — осознанно пропущены
- Files Table → переполнение: пропущено осознанно (короткие имена файлов в нашем продукте)
- Dashboard ActivityFeed → ошибка загрузки: добавим в следующей итерации
```

Это пометки **для следующего аудита** — он не будет задавать те же вопросы и сразу пропустит эти пункты.

## Правила работы

- **По умолчанию только чтение.** Без флага `--fix` директива ничего не меняет в Figma. Это её главное отличие от `final_screens`, `grow_ui_kit`, `component_variants`. Такая природа задачи — сначала видеть картину, потом решать
- **Каждый автофикс — через явный апрув в чате.** Даже в режиме `--fix` директива не правит молча. На каждой находке спрашивает «чинить?» и ждёт ответа
- **Решение «оставить осознанно» — это полноценный результат аудита.** Аудит не должен заставлять чинить всё. Если пользователь сказал «не чинить» — это решение фиксируется в md, и следующий прогон не задаёт тот же вопрос
- **Контекст важен.** Маркетинговый лендинг и админ-форма имеют разные требования к покрытию состояний. Аудит сам понимает по типу экрана, какие проверки применять, какие пропускать
- **Edge cases — отдельная сильная сторона аудита.** Шестая проверка — самое ценное, что даёт директива. Дизайнеры часто забывают состояния, аудит ловит это системно
- **Главный антипаттерн ДС — «почти такие же, но чуть другие» компоненты.** Первая и пятая проверки направлены именно на это. Другие проверки — поддерживающие
- **Дубликаты — это подозрение, не баг.** В третьей проверке аудит помечает кандидатов на объединение, но не решает за дизайнера. Card/Default vs Card/Interactive — два разных смысла, не дубликат. Это всегда продуктовое решение
- **Сохраняй историю прогонов.** Каждый прогон — отдельный файл в `.tmp/`. Это позволяет видеть динамику: что починили, что оставили, что появилось нового
- **Не запускай слишком часто.** Аудит после каждого нового экрана через `final_screens` — норм. Аудит каждые пять минут — бессмысленно, расхождений накопиться не успело
- **Фикс «состояний» — это не задача аудита.** Шестая проверка находит дыры в покрытии, но добавляет состояния `final_screens` отдельным прогоном. Аудит передаёт ему задачу с конкретным брифом
