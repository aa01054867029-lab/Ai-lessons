# Директива: Sitemap → Low-fi Wireframes

## Задача
Превратить подтверждённый sitemap в low-fi wireframes 3-5 ключевых экранов. Сначала — текстовая структура каждого экрана в Markdown, с согласованием на каждом шаге. Отрисовка в Figma Design через `use_figma` — только после финального апрува пользователя.

## Роль
Продуктовый дизайнер с фокусом на layout-проектирование. 5+ лет опыта в SaaS / маркетплейсах / мобильных приложениях. Привычка работать в low-fi до дизайн-системы.

> Адаптируй роль под тип проекта:
> - E-commerce → дизайнер каталога, карточки товара, корзины, чекаута
> - Мобильное приложение → Mobile designer с опытом tab-bar / drawer навигации, привычка к 375px-сетке
> - Лендинг → Marketing-дизайнер, секции от hero до footer
> - B2B SaaS → дизайнер дашбордов, таблиц, сложных форм

## Как устроена директива

Два этапа. Первый — структура каждого экрана в Markdown, апрув после каждого экрана. Второй — отрисовка на Figma Design через `use_figma`, по одному вызову на экран.

Жёсткое правило: **ни одного MCP-вызова к Figma до финального апрува всех текстовых артефактов**. Структурные ошибки ловить в тексте в пятнадцать раз дешевле, чем перерисовывать фрейм.

**Важно про инструмент:** здесь только `use_figma`. `generate_diagram` не подходит — wireframe это layered-фрейм с Auto Layout и секциями, а не диаграмма из Mermaid.

**Важно про файл:** рисуем в Figma **Design**, не FigJam. Дальше по модулю в эти wireframes пойдут нормальные компоненты из UI-кита, это работает только в Design-файле.

## Этап 1. Текстовая сборка

### Фаза 1.1: Выбор экранов

Прочитай `ia/sitemap.md`. Найди все экраны с маркером `[core]`.

Предложи пользователю 3-5 самых критичных для MVP. Если core-экранов больше — выбери те, которые несут основную функциональную нагрузку продукта (обычно это главная рабочая поверхность, экран списка, экран детализации).

Спроси:

> Предлагаю wireframes для: X, Y, Z. Берём этот набор, или заменить?

Не переходи дальше без явного апрува.

### Фаза 1.2: Определение размеров

Спроси:

> Формат — desktop (1440×1024), mobile (375×812) или tablet (768×1024)? Или микс (например, для каждого экрана свой формат)?

Запомни размер для каждого экрана. Размер фрейма задаётся в фазе 2 при отрисовке — здесь фиксируем решение.

### Фаза 1.3: Структура каждого экрана

Для каждого выбранного экрана собери Markdown-описание. Сохраняй по одному файлу в `ia/wireframes/<имя-экрана>.md`. Показывай пользователю, жди апрув, только потом переходи к следующему экрану.

**Формат файла:**

```markdown
# <ScreenName>

**Размер:** desktop 1440×1024 (или mobile 375×812, tablet 768×1024)
**Назначение:** одна строка из sitemap

## Секции (сверху вниз)

### Header (64px высота)
- Logo (левый край)
- Глобальный поиск (центр, 480px)
- Avatar + меню (правый край)

### Sidebar (240px ширина, слева)
- My Files (active)
- Shared
- Recent
- Trash
- Кнопка Upload (внизу)

### Main (всё оставшееся)
- Welcome-блок: «Привет, Даниил» (H1)
- Quick stats (3 карточки в ряд): использовано места / всего папок / всего файлов
- Recent files (список из 5 карточек):
  - Превью-плейсхолдер 48×48
  - Имя файла
  - Дата изменения
  - Размер

## Пустое состояние
Если файлов нет — большая иллюстрация-плейсхолдер по центру + H2 «Пока пусто» + CTA «Загрузить первый файл»

## Placeholder-контент
Используй осмысленный русский контент на основе PRD. Имена файлов: «Презентация Q3.pdf», «Фото с отпуска.jpg». Без lorem ipsum.
```

**Правила структуры:**
- Секции сверху вниз в порядке визуального размещения
- Для каждой секции — размер (если критичен) + содержимое
- Плейсхолдер-контент — осмысленный русский на основе PRD, не lorem ipsum
- Без цветов, иконок, теней, границ — это low-fi
- Если экран имеет empty/loading/error состояния — опиши их отдельной секцией

После каждого файла спроси:

> `<ScreenName>.md` готов. Структура ок? Если надо сдвинуть секции, добавить/убрать — покажи.

Переходи к следующему экрану только после апрува текущего.

### Фаза 1.4: Финальная проверка пакета

Когда все файлы в `ia/wireframes/` подписаны — покажи список:

> Готов к отрисовке: X.md, Y.md, Z.md. Идём на канвас?

## Этап 2. Отрисовка через `use_figma`

Только после апрува всего пакета текстов.

### Фаза 2.0: Получение файла

Запроси ссылку на Figma Design-файл:

> Пришли ссылку на **Figma Design-файл** (не FigJam, не предыдущий файл со sitemap). Лучше пустой или с большим свободным местом — wireframes встанут рядом на одном канвасе.

Извлеки `fileKey` из ссылки.

### Фаза 2.1: Секция для группировки

Создай одну `SectionNode` с именем `Wireframes — low-fi` — в ней будут жить все экраны. Это даёт им общий контейнер и отделяет от того, что появится на файле дальше (UI-кит, финальные экраны).

### Фаза 2.2: Отрисовка экранов — один `use_figma`-вызов на экран

Для каждого файла `ia/wireframes/<screen>.md` — один вызов `use_figma` с JS, который делает следующее:

1. **Загружает шрифты:**
   ```js
   await figma.loadFontAsync({family: "Inter", style: "Regular"});
   await figma.loadFontAsync({family: "Inter", style: "Medium"});
   ```

2. **Создаёт корневой фрейм экрана:**
   ```js
   const frame = figma.createFrame();
   frame.name = "<ScreenName>";
   frame.resize(1440, 1024);  // или 375×812 для mobile
   frame.layoutMode = "VERTICAL";
   frame.primaryAxisSizingMode = "AUTO";
   frame.counterAxisSizingMode = "FIXED";
   frame.paddingTop = frame.paddingBottom = 0;
   frame.paddingLeft = frame.paddingRight = 0;
   frame.itemSpacing = 0;
   frame.fills = [{type: "SOLID", color: {r: 1, g: 1, b: 1}}];
   ```

3. **Для каждой секции — вложенный фрейм с Auto Layout:**
   ```js
   const section = figma.createFrame();
   section.name = "Header";
   section.layoutMode = "HORIZONTAL";   // или VERTICAL, по смыслу
   section.primaryAxisSizingMode = "FIXED";
   section.counterAxisSizingMode = "FIXED";
   section.resize(1440, 64);
   section.paddingLeft = section.paddingRight = 32;
   section.itemSpacing = 24;
   section.counterAxisAlignItems = "CENTER";
   frame.appendChild(section);
   ```

4. **Плейсхолдеры-картинки — прямоугольники с заливкой `#E5E5E5`:**
   ```js
   const ph = figma.createRectangle();
   ph.resize(48, 48);
   ph.fills = [{type: "SOLID", color: {r: 0.898, g: 0.898, b: 0.898}}];  // #E5E5E5
   ph.name = "Placeholder";
   section.appendChild(ph);
   ```

5. **Текстовые плейсхолдеры — `createText` с Inter Regular для body, Inter Medium для заголовков:**
   ```js
   const t = figma.createText();
   t.fontName = {family: "Inter", style: "Medium"};
   t.fontSize = 20;
   t.characters = "Привет, Даниил";
   t.fills = [{type: "SOLID", color: {r: 0.1, g: 0.1, b: 0.1}}];
   section.appendChild(t);
   ```

6. **Добавляет фрейм в `SectionNode`** из фазы 2.1:
   ```js
   wireframesSection.appendChild(frame);
   ```

**Правила отрисовки:**
- Все фреймы секций — Auto Layout. Это важно для дальнейшей ручной правки.
- Именование слоёв — на латинице по смыслу: `Header`, `Sidebar`, `Main`, `RecentFilesList`, `RecentFileItem`. Без русского, без UUID, без `Frame 123`.
- Цвета: только `#FFFFFF` (фон корневого фрейма), `#E5E5E5` (плейсхолдеры картинок), `#1A1A1A` (текст). Никаких брендовых цветов — это будет в ДС.
- Шрифт — только Inter Regular / Medium. Никакой типографики бренда.
- Экраны раскладывай по горизонтали внутри `SectionNode`: по оси X с gap 80px. Заголовок-текст над каждым фреймом («Dashboard», «Files», «FileDetails»).

### Фаза 2.3: Проверка после write

После каждого вызова `use_figma`:
1. Вызови `get_screenshot` на созданном фрейме и покажи пользователю.
2. Если Auto Layout развалился (бывает при вложенных секциях) — покажи это сразу, не переходи к следующему экрану без подтверждения.

### Фаза 2.4: Финальный показ

Покажи пользователю:

> Wireframes на канвасе:
> - X — <ссылка на фрейм>
> - Y — <ссылка на фрейм>
> - Z — <ссылка на фрейм>
>
> Тексты лежат в `ia/wireframes/`. Если правишь структуру — правь в `.md`-файле и перезапускай отрисовку конкретного экрана, не редактируй вручную на канвасе: уйдёт рассинхрон, на следующем прогоне откатится.

## Формат результата

В папке проекта после прогона директивы:

```
ia/
  wireframes/
    dashboard.md          — структура экрана
    files.md
    filedetails.md
```

Плюс SectionNode `Wireframes — low-fi` в Figma Design-файле, внутри — 3-5 фреймов-экранов с Auto Layout и серыми плейсхолдерами.

> Адаптируй output под тип проекта:
> - E-commerce → обязательно включи Product / Catalog / Cart / Checkout
> - Мобильное приложение → проверь, что tab-bar / drawer единообразны на всех экранах
> - Лендинг → один длинный фрейм с секциями, а не несколько экранов
> - B2B SaaS → особое внимание к табличным представлениям и формам с большим количеством полей

## Правила работы

- **Никаких MCP-вызовов до апрува всех текстовых артефактов.** Главное правило. Текстовые правки бесплатны, перерисовка фрейма — MCP-вызов и время.
- **Ни одной фазы без явного подтверждения.** Если пользователь молчит или отвечает неопределённо — переспроси: «апрувим и идём дальше, или есть правки?»
- **Low-fi значит low-fi.** Никаких цветов бренда, иконок, теней, градиентов, скруглений больше 4px. Серые прямоугольники, Inter, 3 цвета. Цвета и типографика придут из ДС в блоке C.
- **Placeholder-контент — осмысленный.** Имена файлов, названия папок, заголовки — из PRD. Lorem ipsum и «Text 1 / Text 2» запрещены: на ревью с командой не на что опереться.
- **Имена слоёв — на латинице по смыслу.** `Header`, `Sidebar`, `MainContent`, `RecentFilesList`. Это задел для работы с ДС в следующих уроках.
- **Auto Layout везде, где секция может растянуться.** Проверяй после отрисовки: измени контент одной карточки — перестраивается ли ряд. Если нет — пересобирай секцию.
- **Экран не должен выходить за границы фрейма.** Если контент больше — либо увеличивай высоту корневого фрейма, либо дроби контент на вертикальные секции со скроллом.
- **Рассинхрон канваса и текста — твоя зона ответственности.** Если пользователь правит что-то на канвасе руками — предупреди, что следующий прогон директивы откатит правки.
