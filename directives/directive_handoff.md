# Директива: Передача проекта на GitHub (handoff)

> **Запуск:** только когда пользователь явно дал эту директиву. Переиспользуемая: первый запуск создаёт репозиторий и заливает проект, повторные — вносят изменения. Пуш делает агент ОДНОЙ операцией (не по файлу); ученик командную строку не пишет, только запускает директиву и проходит шаги, которые директива выводит.

## Задача
Отдать проект разработке через GitHub: залить ОДНИМ пушем и дать ссылку. Директива выводит пользователю пошагово, что нужно (создать репозиторий и токен), затем заливает выбранную часть проекта в main и создаёт README. На выходе — HTTPS-ссылка для разработчика.

## Когда применять
- После `storybook`: есть законченный артефакт (база + каталог).
- Переиспользуемая: репозиторий уже есть → вносим изменения (в отдельную ветку); см. Фазу 5.

## Предусловие
1. Аккаунт на GitHub.
2. **GitHub подключён к Claude** на claude.ai: Customize → Connectors → поиск GitHub → GitHub integration → Connect → авторизация → вернуться в проект и **перезагрузить чат** (или перезапустить VS Code; при новых подключениях чат нужно перезагрузить).
3. `git` в системе (агент проверяет `git --version`).

## Ключевой принцип
Пуш — **одна операция** (`git`-коммит и `git push` всего выбранного сразу), не по файлу: пофайловая заливка по API — это полчаса ожидания, так НЕ делаем. Авторизация — **classic-токеном** (его создаёт пользователь по шагам, что выводит директива). Токен транзиентный: в файлы и `.git/config` не попадает. Командную строку ведёт агент; ученик проходит только браузерные шаги (репозиторий, токен, коннектор).

## Фаза 0. Вывести шаги и проверки
1. `git --version` — есть.
2. Выведи пользователю ДВА шага (подробно), без них продолжить нельзя:
   - **Создать пустой репозиторий:** github.com → аккаунт → Repositories → New → имя латиницей → Private → **без** README/.gitignore/license → Create. Скопировать имя/ссылку.
   - **Создать classic-токен:** иконка профиля → Settings → Developer Settings → Personal Access Tokens → Tokens (classic) → Generate new token (classic) → название (любое) → срок: **No expiration** → отметить **все scopes** (чтобы был доступ и к приватным репо) → Generate → **скопировать один раз** (повторно не покажется).
3. Попроси прислать имя репозитория и токен. **Токен нигде не сохраняй и в файлы не пиши** — он нужен только для пуша.

## Фаза 1. Что заливаем
Спроси, что заливать: **весь проект / только React-база / только прототип**. Залей выбранное. В любом случае исключи служебное: `node_modules/`, `dist/`, `build/`, `.env*`, `.tmp/`, `.DS_Store` (через `.gitignore`, создай при отсутствии).

## Фаза 2. Подготовить и запушить (одной операцией)
1. `git init` если нужно; ветка `main` (`git branch -M main`); `git add -A` → `git commit -m "<проект: что заливаем>"`.
2. Remote: чистый `git remote add origin https://github.com/<user>/<repo>.git`.
3. **Один пуш** с токеном по HTTPS, не оставляя токен в конфиге, например разовой командой: `git push https://<TOKEN>@github.com/<user>/<repo>.git main`. Весь выбранный объём уходит за секунды одним коммитом. Пофайлово — нельзя.

## Фаза 3. README и запись в среду
1. Создай `README.md` в корне: что за проект, из чего состоит (база, каталог), как запустить. На GitHub он показывается на странице репозитория — хороший тон.
2. Создай/обнови `PROJECT.md`: ссылка на репозиторий, ветка `main` — чтобы следующие директивы (деплой) знали репозиторий. Закоммить и запушь тем же способом.

## Фаза 4. Проверить и отдать
1. Обнови страницу репозитория: файлы на месте, ветка main, README выводится, `node_modules` НЕ залит.
2. Дай пользователю HTTPS-ссылку — её копируют и отправляют разработчику.

## Фаза 5. Повторный запуск = обновление
Репозиторий уже есть. Если правки в основную линию — `git add -A` → `git commit` → один `git push` в main. Если это незаконченная итерация — заливай в **отдельную ветку** (как делают с фичами), не трогая main. Новый репозиторий не создаём, `PROJECT.md` держим актуальным. В финале перечисли, что обновил и в какую ветку.

## Self-check (перед финалом)
- Пуш прошёл ОДНОЙ операцией, не пофайлово?
- `node_modules`/`dist`/`.env` НЕ в репозитории?
- Токен НЕ попал в `.git/config`, коммиты, `PROJECT.md` или любой файл?
- README на месте; `PROJECT.md` со ссылкой записан?
- Ссылка открывается, ветка main.

## Финал (вывести пользователю)
```
Проект передан на GitHub.
- Репозиторий: <ссылка> · ветка: main
- Залито одним пушем: <весь проект / React-база / прототип> (node_modules исключён)
- README + запись в среде (PROJECT.md)
Копируешь HTTPS-ссылку и отдаёшь разработчику.
```

## Грабли (учтено)
- **Пофайловая заливка по API = ~30 минут.** Главная грабля. Только одна операция пуша — секунды.
- **Токен.** Classic, все scopes, без срока — иначе агент не достучится до приватного репо. Копируется один раз. В файлы/`.git/config` НЕ попадает (только в команду пуша).
- **Подключение GitHub integration** на claude.ai → после него обязательно перезагрузить чат / перезапустить VS Code, иначе Claude Code не подхватит.
- **Репозиторий не пустой** (с README) → конфликт первого пуша. Просим пустой; если уже с README — `git pull --rebase` перед пушем.
- **`node_modules` в репозитории** — тяжесть. В `.gitignore` до `git add`.
- **Сломанный OAuth `/mcp` github** («does not support dynamic client registration») — это про отдельный MCP-сервер в Claude Code, НЕ про claude.ai GitHub integration. На него не опираемся; передача идёт через integration + токен.
- **CLI пугает дизайнера** — но команды ведёт агент; ученик проходит только браузерные шаги.

## Правила
- Один пуш всего выбранного; пофайлово — нет.
- Репозиторий и токен создаёт пользователь по шагам, что выводит директива.
- Токен транзиентный: для пуша, не в файлы.
- Служебное (`node_modules`, `dist`, `.env`) не попадает.
- Повторный запуск — обновление того же репозитория (main или отдельная ветка).
- Командную строку ведёт агент; ученик — только браузерные шаги.
