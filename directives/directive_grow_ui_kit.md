# Директива: Расширение UI-кита (grow_ui_kit)

## Задача
Добавить в существующую дизайн-систему недостающие компоненты, привязанные к тем же Variables, что и стартовый набор. На входе — `ds/components.md` после `ds_baseline` или `ds_scan`. На выходе — обновлённый `ds/components.md` с новыми строками плюс компоненты в SectionNode «UI Kit — extended» рядом с «Foundation» в том же Figma-файле. Финал — короткий аудит на консистентность с baseline.

## Роль
Дизайн-системный архитектор с опытом расширения UI-китов. Знает, какие компоненты пора заводить в ДС, а какие достаточно собрать композицией существующих. Привычка проверять каталог перед заведением нового, чтобы не плодить дубликаты.

> Адаптируй роль под продукт:
> - B2C веб → дизайнер с фокусом на формы заказа, карточки товара, модальные окна подтверждения
> - B2B SaaS → дизайнер дашбордов, сложные таблицы, фильтры, контекстные меню
> - Мобильное приложение → дизайнер с фокусом на тач-таргеты, тонкая работа с Toast и Sheet
> - Лендинг → дизайнер маркетинговых страниц, hero-секции, accordion, sticky-CTA

## Как устроена директива

Четыре фазы. Первая — чтение существующего `ds/components.md` плюс предложение списка типовых пробелов. Вторая — md-описания новых компонентов с апрувом. Третья — пакетная отрисовка через `use_figma`. Четвёртая — запись в `ds/components.md` плюс короткий аудит.

Жёсткое правило: **никаких MCP-вызовов до апрува текстовых описаний**. Та же логика, что в `ds_baseline`. Структурные правки в md дешевле перерисовки фрейма.

**Важно про инструмент:** `use_figma` для записи на канвас, `get_variable_defs` (или чтение через `figma.variables.getLocalVariablesAsync()`) для проверки привязок при аудите. Прибитые hex запрещены — это инвариант контракта `ds/CONTRACT.md`.

**Важно про файл:** работаем в том же Figma Design-файле, где лежат wireframes и Foundation. SectionNode «UI Kit — extended» создаём рядом с «Foundation», не внутри — чтобы стартовые компоненты остались как ядро.

## Фаза 1. Чтение каталога и выбор компонентов

### Фаза 1.1: Чтение `ds/components.md`

Прочитай существующий каталог. Выпиши имена всех компонентов, чтобы:
- Не предложить дубль
- Понять, какие смысловые слои уже покрыты (действия, ввод, контейнеры, навигация, фидбэк, данные)

Если файл отсутствует или пуст — переспроси: «`ds/components.md` пустой. Сначала нужен `ds_baseline` или `ds_scan`. Запустить?»

### Фаза 1.2: Предложение списка

Покажи пользователю список «обычно недостающих после baseline» с короткими пояснениями:

```
Что обычно добавляют после baseline:

1. Table       — список с заголовками, плотностями, состояниями
2. Tabs        — переключение между разделами на одном экране
3. Tooltip     — короткая подсказка над элементом
4. Dropdown    — триггер плюс панель опций или меню
5. Select      — поле выбора одного значения из списка
6. Toast       — всплывающее уведомление (success/warning/error/info)
7. Breadcrumb  — путь по иерархии страниц
8. Pagination  — постраничная навигация по списку
```

Спроси:

> Какие из этих добавляем? Можно убрать ненужные, добавить своё (имя + одна строка назначения). Бери только то, чего реально не хватает на твоих экранах — UI-кит лучше держать компактным.

Зафиксируй итоговый набор. Если пользователь добавляет компонент не из списка — переспроси, нет ли уже близкого в `ds/components.md`. Перед заведением нового — чек на дубликаты.

## Фаза 2. Markdown-описания

Для каждого компонента собери описание в одном файле `ds/.tmp/grow_ui_kit_draft.md`. Это черновик — после апрува содержимое уйдёт в `ds/components.md` строками.

**Шаблон одного компонента:**

```markdown
### <ComponentName>

**Назначение:** одна строка, что это в продукте.

**Анатомия:**
- <слой 1> (например, Header)
- <слой 2> (например, Row)
- <слой 3> (например, Cell)

**Варианты:**
- <variant key>: <val1> | <val2> | <val3>
- <variant key>: <val1> | <val2>

**Состояния:**
- default / hover / disabled (или специфичные для компонента — selected, loading, focus)

**Отступы:** только из шкалы Primitive (space-1 … space-16). Без значений вне токенов.

**Привязки к Variables:**
- fills → surface-* / bg-*
- strokes → border-*
- text → text-*
- typography → font-sans, size-*, weight-*
```

Покажи всё описание разом. Дождись апрува по каждому компоненту. Если по одному компоненту правка — внеси и переспроси.

**Минимальная матрица вариантов на старте.** Не плодить шесть variants со всеми состояниями сразу — write-tools в бете на больших матрицах сбоят. Два-три варианта на компонент. Расширение матрицы вариантов и состояний (hover/disabled/loading) — отдельная директива поверх этой.

## Фаза 3. Отрисовка через `use_figma`

### Фаза 3.0: Получение файла

Если `fileKey` уже сохранён из предыдущей директивы (`ds_baseline` или `ds_scan`) — используй его. Если нет — переспроси:

> Пришли ссылку на тот же Figma Design-файл, где лежит SectionNode «Foundation». Новый блок «UI Kit — extended» поставим рядом, в том же файле.

### Фаза 3.1: Создание SectionNode

Если SectionNode «UI Kit — extended» уже существует (повторный прогон) — используй его, не создавай дубль. Если нет — создай рядом с «Foundation»:

```js
const ext = figma.createSection();
ext.name = "UI Kit — extended";
// Расположи справа от Foundation с зазором 80
```

### Фаза 3.2: Пакетная отрисовка

Группируй компоненты по 2-3 на один вызов `use_figma`. Не пытайся все 6-8 одним вызовом — write-tools на длинных пакетах теряют контекст и сбоят на привязках к Variable.

**Правила отрисовки:**

1. **Компоненты — это настоящие `ComponentNode` / `ComponentSetNode`, не `FrameNode`.** Тот же инвариант, что в `ds_baseline`. Создавать через `figma.createComponent()` или `figma.createComponentFromNode(frame)`. Варианты — через `figma.combineAsVariants([...components], parent)` с именами в формате `Property=value` (например `Density=compact`, `State=hover`). Frame-«компоненты» не видны в Assets-панели и непригодны для `createInstance()` на финальных экранах.

2. **Типографика — только через TextStyles, созданные в `ds_baseline`.** Найти их через `getLocalTextStylesAsync()` (учитывая нормализацию пробелов вокруг `/` в имени), применять через `await text.setTextStyleIdAsync(style.id)`. **НЕ создавать новые TextStyles в этой директиве** — UI-кит должен использовать ту же шкалу, что и Foundation. Если нужного стиля не хватает — добавь сначала в `ds_baseline`/`foundation.md`, потом возвращайся сюда. Прибивать `fontSize`/`fontName` напрямую запрещено.

3. **Все fills и strokes — через Variables.**
   ```js
   const allVars = await figma.variables.getLocalVariablesAsync();
   const surfaceDefault = allVars.find(v => v.name === "surface-default");
   const paint = figma.variables.setBoundVariableForPaint(
     {type: "SOLID", color: {r:0,g:0,b:0}}, "color", surfaceDefault
   );
   row.fills = [paint];
   ```

4. **Радиусы — через Variables** (четыре угла отдельно: `topLeftRadius`, `topRightRadius`, `bottomLeftRadius`, `bottomRightRadius`). **Отступы и `itemSpacing`** — биндить через Variables к `space-*`, не прибивать числа.

5. **Auto Layout везде.** Любая секция, любой компонент. Без исключений.

6. **Имена компонентов и variants — латиницей.** Имя ComponentSet — короткое (`Table`, `Tabs`, `Toast`), variants — формата `Property=value` (`Density=compact`, `Density=default`; `State=default`, `State=hover`). `combineAsVariants` собирает из этого правильную property-схему в Properties-панели.

7. **Раскладка внутри SectionNode «UI Kit — extended»:** компоненты идут в одном wrapping-фрейме (`layoutMode = "HORIZONTAL"`, `layoutWrap = "WRAP"`). Между плитками `itemSpacing = 32`. Над каждой плиткой — лейбл `Label/xs` с именем компонента (`text-muted`).

8. **Никакого `throw` в конце мутирующего вызова.** `use_figma` исполняет каждый вызов как транзакцию: `throw` в конце откатывает все мутации того же вызова (`createComponent`, `combineAsVariants`, `setTextStyleIdAsync`, `setBoundVariable*` — всё). Хочешь отчёт — два вызова: первый мутирует (заканчивается естественно), второй читает результат и кидает throw с метрикой. Альтернатива — клади статус в `node.name`. Симптом нарушения: «лог говорит applied=N, в файле visually ничего не изменилось».

### Фаза 3.3: Проверка после write

После каждого мутирующего пакета — **отдельным** read-only вызовом:
1. `findOne(n => n.type === 'COMPONENT_SET' && n.name === '<Имя>')` — должен вернуть узел нужного типа. Если null или type='FRAME' — компонент не создан, фикси перед следующим пакетом.
2. Пройдись по всем text-узлам в созданном компоненте: `t.textStyleId` должен быть непустым. Если есть пустые — `setTextStyleIdAsync` откатился (см. правило 8 выше).
3. Сверь fill через `node.boundVariables.fills` — должен быть массив с VARIABLE_ALIAS. Если SOLID без bound — пересобирай этот компонент.
4. `get_screenshot` на созданной секции, покажи пользователю.

## Фаза 4. Запись в каталог и аудит

### Фаза 4.1: Дописать в `ds/components.md`

Каждый новый компонент — отдельной строкой в разделе по смыслу (Действия / Ввод / Контейнеры / Навигация / Фидбэк / Данные). Формат строки тот же, что в `ds_scan`:

```markdown
### Table
- **Назначение:** табличный список с заголовками, плотностями и состояниями строк
- **Варианты:** density: compact | default; row-state: default | hover | selected | loading; sticky-header: yes | no
- **Node ID:** `1:512`
```

Покажи обновлённый `ds/components.md`. Если в нём появились дубликаты (компонент с таким именем уже был) — пометь это и попроси решение: переименовать новый или удалить старую запись.

### Фаза 4.2: Короткий аудит

Один MCP-вызов после записи. Запрос (внутренний для агента, пользователю показываем результат):

```
Сверь компоненты SectionNode «UI Kit — extended» с компонентами SectionNode «Foundation» на:
1. Типографика — везде ли через Variables (font-sans, size-*, weight-*)
2. Цвета — все ли fills/strokes привязаны к Semantic, нет ли прибитых hex
3. Отступы — все ли значения из шкалы space-* (без 14, 18, 22 и других не-токенных значений)

Отчёт построчно: каждый новый компонент → ✓ или конкретная проблема.
```

Покажи отчёт пользователю. На каждое расхождение — короткое предложение правки. Если расхождение по отступу (не из шкалы) — поправь руками или переотрисуй через `use_figma`.

## Правила работы

- **Никаких MCP-вызовов до апрува md-описаний.** Главное правило. Структурные правки в тексте бесплатны, перерисовка пакета компонентов — несколько `use_figma`-вызовов и ощутимое время.
- **Прибитые hex запрещены.** Это инвариант контракта. Если поймал в результате — переотрисовка, не «допишу руками потом».
- **Минимальная матрица вариантов на старте.** Два-три варианта на компонент, не больше. Шесть variants со всеми состояниями ломаются на write-tools. Расширение — отдельная директива.
- **Чек на дубликаты перед заведением.** Прочитал `ds/components.md` → понял, что есть → не плодим близнеца. Если близкое уже есть — расширяем variants существующего, не заводим новый.
- **Один Figma-файл на проект.** SectionNode «UI Kit — extended» рядом с «Foundation» и «Wireframes — low-fi». Не плодим файлы.
- **Пакеты по 2-3 компонента на `use_figma`-вызов.** Дольше пакеты — больше шанс, что развалится Auto Layout или отвалится привязка к Variable.
- **Аудит обязателен.** Без шага «сверь с baseline» расхождения копятся. Один MCP-вызов в конце окупает себя на следующих десятках компонентов.
- **Имена слоёв — латиницей.** Имя ComponentSet — `Table`, `Tabs`, `Toast`. Имя variant — `Property=value` (`Density=compact`, `State=hover`). Не русские, не `Frame 234`.
- **Структурные правки — в `ds/components.md`, не на канвасе.** Если правишь компонент руками в Figma — обновляй индекс. Если правишь индекс — перезапускаешь фазу 3 директивы. Расхождение между Figma и md ломает последующие сборки экранов.
