# Директива: Выпуск AI-среды — навыки, субагенты, оркестрация (release_environment)

> **Запуск:** только когда пользователь явно дал эту директиву. Выполняешь по шагам, ничего не додумываешь сверх неё. Источник истины об архитектуре — ЖИВАЯ документация Anthropic (читаешь при запуске, см. Фазу 0), а о содержании среды — директивы проекта и локальная история работы пользователя с агентом.

## Задача
Свернуть накопленное за работу — директивы и реальный опыт работы с агентом — в полноценную AI-среду по актуальной архитектуре Anthropic: **навыки** (`.claude/skills/`), **субагенты** (`.claude/agents/`), **оркестрацию** (`CLAUDE.md`) и структуру. Среда собирается не из головы, а из того, как пользователь реально работал. Старое архивируется, не удаляется. На выходе — собранная среда и сводка миграции.

Это не «придумать среду с нуля». Это перенос: директивы, которые вызывали руками, становятся навыками, срабатывающими по описанию; повторяющиеся узкие задачи — субагентами; правила работы — оркестрацией.

## Когда применять
- Накопились директивы и история работы с агентом по проекту, и пора собрать из них рабочую среду.
- Designer = read-not-write: ты направляешь и читаешь результат, среду собирает агент.
- Запускается, когда есть что сворачивать. Повторный запуск — режим **обновления** (см. Фазу 7): добавить новое, обновить устаревшее, не затирать рабочее.

## Предусловие
- Проект с историей: директивы проекта (папка `directives/` или аналог) и/или локальная история чатов Claude Code.
- Доступ в интернет — директива читает актуальную документацию (Фаза 0). Если сети нет, останови и скажи пользователю: собирать архитектуру по памяти нельзя, она устаревает.

## Ключевой принцип
**По живым докам и по простоте.** Как устроены навыки/субагенты/оркестрация, Anthropic периодически пересматривает — бери архитектуру из актуальной документации при каждом запуске, не из зашитого снимка. И собирай под то, как пользователь реально работает (из истории), а не по максимуму: добавляй механику только там, где она улучшает результат.

## Фаза 0. Сверка с актуальной архитектурой (живые доки)
1. Прочитай актуальную документацию Anthropic (через WebFetch/браузер). Целевые страницы (могли переехать — если 404, найди актуальные поиском по сайту):
   - Agent Skills, обзор: `platform.claude.com/docs/en/agents-and-tools/agent-skills/overview`
   - Навыки в Claude Code: `code.claude.com/docs/en/skills`
   - Субагенты: `code.claude.com/docs/en/sub-agents`
   - Память и CLAUDE.md: `code.claude.com/docs/en/memory`
   - Принципы построения агентов: `anthropic.com/research/building-effective-agents`
2. Зафиксируй из доков текущие правила (на момент запуска):
   - **Навык** — папка `.claude/skills/<name>/` с файлом `SKILL.md`; во frontmatter `name` + `description` (description описывает, ЧТО делает и КОГДА применять — это триггер срабатывания); тело подгружается при срабатывании, ресурсы по мере нужды (прогрессивная подгрузка).
   - **Субагент** — файл в `.claude/agents/` (frontmatter + промпт); своё окно контекста, свои инструменты/права; возвращает наверх только итог; подхватывается при старте.
   - **Оркестрация** — `CLAUDE.md` (правила главного агента), грузится в каждую сессию; держать коротким.
   - **Принцип простоты** — усложнять только когда это реально улучшает результат.
3. Если доки недоступны — останови выполнение и сообщи пользователю.

## Фаза 1. Инвентаризация входа
1. Собери директивы проекта: список и по каждой кратко — что делает, как вызывалась.
2. Прочитай локальную историю работы с агентом: `~/.claude/projects/<папка-проекта>/*.jsonl` (папка проекта в этом пути соответствует текущему проекту; вне git берётся по корню). Пройди по сессиям и суммаризируй: какие задачи решались снова и снова, какими приёмами, какие поправки/правила пользователь повторял.
3. По итогу выдели кандидатов:
   - **в навыки** — повторяющиеся задачи и приёмы (включая то, что закрывали директивы);
   - **в субагенты** — узкие задачи, которые только читают/собирают, независимы и засоряли основной контекст (широкий поиск, сбор данных, аудит).
4. Покажи пользователю короткий план: что станет навыком, что субагентом, что войдёт в оркестрацию. Дальше собираешь по нему.

## Фаза 2. Навыки
Для каждого устойчивого паттерна и директивы:
1. Заведи навык `.claude/skills/<name>/SKILL.md`. Frontmatter `name` + `description` (что делает И когда применять — без размытости, иначе навык не сработает). Тело — шаги из директивы/паттерна. Сопутствующие файлы/скрипты положи в папку навыка.
2. Директивы как жанр уходят: их работу забирают навыки, срабатывающие по описанию. Содержание директивы сохрани в навыке (не потеряй шаги и грабли).

## Фаза 3. Субагенты
1. Заведи субагента `.claude/agents/<name>.md` (frontmatter + промпт, ограниченный набор инструментов) только для задач, которые: только чтение/сбор + независимы + иначе засоряют основной контекст.
2. НЕ заводи субагентов под интерактив (Figma, браузер) — это делает основной агент. И под линейную/пишущую работу субагент не нужен, он только добавит задержку и расход.

## Фаза 4. Оркестрация и структура
1. Собери `CLAUDE.md`: роль главного агента, когда какой навык поднимать, что отдавать какому субагенту, базовые правила проекта.
2. Держи его коротким (ориентир ≤200 строк) — он грузится в каждую сессию. Длинное и ситуативное выноси в навыки или в path-scoped правила (`.claude/rules/`). Убери противоречия — на конфликте правил агент выберет произвольно.

## Фаза 5. Архивация старого
1. Старые директивы и заменённые файлы перенеси в `_archive/` (с пометкой даты/причины). **Не удаляй** — чтобы можно было откатиться и свериться.

## Фаза 6. Сводка миграции (вывести пользователю)
Короткий отчёт: что стало навыком, что субагентом, что вошло в оркестрацию, что заархивировано и как теперь пользоваться средой.

## Фаза 7. Повторный запуск = обновление
Если среда уже собрана и директива запущена снова: не пересобирай с нуля. Перечитай свежие доки (Фаза 0) и свежую историю, добавь новые навыки/субагентов, обнови устаревшее под актуальную архитектуру, заменённое — в `_archive/`. Рабочее не затирай. В финале перечисли, что изменил.

## Self-check (перед финалом)
- Архитектура взята из живых доков этого запуска, а не по памяти?
- У каждого навыка `description` говорит и ЧТО, и КОГДА (триггер не размыт)?
- `CLAUDE.md` в пределах ориентира по длине, без противоречий; длинное вынесено в навыки/rules?
- Субагенты только под чтение/сбор/независимое; интерактив и линейная/пишущая работа оставлены основному агенту?
- Старое заархивировано, не удалено?
- Сводка миграции выведена пользователю?

## Финал (вывести пользователю)
```
AI-среда собрана по актуальной архитектуре Anthropic.
- Навыки: <N> → .claude/skills/ (срабатывают по описанию)
- Субагенты: <M> → .claude/agents/
- Оркестрация: CLAUDE.md (<строк> строк)
- Заархивировано: <что> → _archive/
Как пользоваться: говоришь, что нужно — навык поднимается сам; новый навык/субагент подхватится после перезапуска.
Собрано из: <живые доки + история работы / только директивы — история недоступна>
```

## Грабли (учтено)
- **Архитектура устаревает** — Anthropic меняет, как должны быть устроены навыки/субагенты. Читай живые доки при каждом запуске; нет сети — остановись, не собирай по памяти.
- **Локальная история** — путь `~/.claude/projects/<папка-проекта>/*.jsonl`. Если пусто/недоступно — собери из директив и скажи пользователю, что историю не прочитал.
- **CLAUDE.md раздувается** — он грузится в каждую сессию целиком; чем толще, тем хуже соблюдается. Держи коротким, лишнее — в навыки/rules.
- **Размытое описание навыка** — навык не подхватится, когда нужен. В `description` пиши и что делает, и когда применять.
- **Субагент под интерактив** — Figma/браузер и подобное живое подключение в отдельное окно субагента не дотягивается; это основной агент. И субагент не видит твою переписку — давай ему самодостаточную задачу.
- **Новый навык/субагент не виден** — подхватывается после перезапуска среды.
- **Удаление вместо архива** — ничего не удаляй; старое в `_archive/`, чтобы был откат.
- **Доступы/токены** — эта директива их не трогает и никуда не записывает.

## Правила
- Designer = read-not-write: направляешь и смотришь результат, среду собирает агент.
- Простота: собирать под реальную работу из истории, не по максимуму.
- Архитектуру брать из живой документации Anthropic, не из памяти.
- Старое архивируется, не удаляется.
