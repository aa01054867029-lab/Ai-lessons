# Директива: Публикация и деплой (deploy)

> **Запуск:** только когда пользователь явно дал эту директиву. Переиспользуемая: первый запуск публикует, повторные — обновляют (пересборка и перезаливка/пуш). Командную строку и подключение ведёт агент; ученик проходит браузерные шаги (для сервиса) или отдаёт доступ к серверу (для своего VPS).

## Задача
Выложить рабочий прототип в сеть и дать публичную ссылку. Два пути на выбор: **готовый сервис** (Vercel / Netlify / Cloudflare Pages) или **свой VPS** (тот же сервер, что арендован под VPN; открытие по IP). Директива спрашивает путь, готовит проект к деплою, выполняет публикацию и записывает живую ссылку в `PROJECT.md`.

## Когда применять
- После оживления (`wire`): прототип рабочий локально, поднимается `npm run dev`.
- Нужна публичная ссылка — для юзер-теста, показа, передачи.
- Переиспользуемая: обновил продукт → запускаешь снова, она перезаливает изменения тем же путём.

## Предусловие
- Проект собирается: `npm run build` без ошибок (Vite → папка `dist`).
- **Путь «сервис»:** GitHub-репозиторий (из `handoff`) + GitHub подключён к Claude. Важно: на репозитории может лежать только компонентная база (если `handoff` запускали до сборки экранов) — актуальный прототип нужно будет дозалить (Фаза A1).
- **Путь «свой сервер»:** доступ к VPS — **IP, пользователь (обычно root), пароль**. То, что выдал провайдер при аренде.

## Ключевой принцип
Деплой статики Vite — это сборка (`dist`) + отдача её веб-сервером/CDN. Главное, что нужно настроить в обоих путях, — **SPA-маршрутизация**: любой неизвестный путь отдаёт `index.html`, иначе прямые ссылки на внутренние экраны после деплоя дают 404. Секреты и доступы **транзиентные**: пароль сервера и токены — только для операции, не в файлы, не в `.git/config`, не в `PROJECT.md`.

---

## Фаза 0. Выбор пути и общая подготовка
1. Спроси: **куда деплоим — на сервис или на свой сервер?** Дальше иди по выбранной ветке.
2. Общая подготовка (для обоих):
   - `npm run build` — проект собирается без ошибок; на выходе `dist`.
   - **SPA-маршрутизация:** настрой fallback на `index.html` (как именно — зависит от пути, см. ниже).
   - `vite.config` `base`: для отдачи из корня домена/порта оставь `/` (для своего сервера по IP это корень — `/`).
   - Служебное в `.gitignore`: `node_modules/`, `dist/`, `.env*`. Секреты — в переменные окружения, не в код.

## Путь A — готовый сервис (Vercel / Netlify / Cloudflare)

### Фаза A1. Обновить репозиторий
Сервис собирает сайт из репозитория, поэтому в нём должна лежать **актуальная версия прототипа** (а не только база с урока передачи).
1. Проверь, что в репозитории текущий проект. Если устарел — залей актуальную версию **одним пушем в main** (тем же способом, что `handoff`: `git add -A` → `commit` → `git push`). Токен транзиентный, в конфиг не пишем.
2. Убедись, что `node_modules`/`dist`/`.env` не уходят (см. `.gitignore`).

### Фаза A2. Подключить сервис (браузерные шаги — выведи пользователю)
SPA-маршрутизация для сервиса: положи в проект конфиг fallback (Vercel — `vercel.json` с rewrite всех путей на `/index.html`; Netlify — `_redirects` со строкой `/* /index.html 200`; Cloudflare Pages — то же `_redirects`). Закоммить и запушь.
- **Vercel (по умолчанию):** vercel.com → войти через GitHub → Add New → Project → Import выбранного репозитория → фреймворк определится (Vite) автоматически → Deploy.
- **Netlify:** Add new site → Import an existing project → выбрать репозиторий → Deploy.
- **Cloudflare Pages:** Create application → Pages → Connect to Git → выбрать репозиторий → Deploy.

### Фаза A3. Сделать публичной, проверить и записать
1. **Vercel: убрать стену авторизации (браузерный шаг — выведи пользователю).** По умолчанию Vercel закрывает деплой своей авторизацией (Vercel Authentication) — чужой по ссылке упрётся в логин. Проект → Settings → **Deployment Protection** → выключить **Vercel Authentication** → Save. (Netlify / Cloudflare Pages публичны по умолчанию — этого шага нет.)
2. **Копируй стабильную ссылку, не на конкретный билд.** У каждого деплоя есть свой одноразовый URL билда; для теста и передачи бери постоянную ссылку проекта (Production Domain на странице проекта) — на ней сами отображаются все новые апдейты.
3. Получи ссылку, открой её. Проверь внутренние роуты: прямой путь на внутренний экран не должен давать 404 (проверка SPA-fallback).
4. Авто-пересборка: на каждый пуш в `main` сервис пересобирает сам (~2–3 мин). Совет ученику: пока разрабатываешь — смотри на локалку (моментально), а живую версию обновляешь пушем, когда устраивает.
5. Запиши **стабильную** ссылку в `PROJECT.md` (Фаза финал).

## Путь B — свой VPS (открытие по IP)

### Фаза B1. Получить доступ
Запроси у пользователя: **IP-адрес, пользователь (root), пароль** (то, что выдал провайдер). **Доступ транзиентный:** используется только для подключения, НЕ пишется в файлы, `.git/config`, `PROJECT.md`, историю команд по возможности.

### Фаза B2. Залить и поднять отдачу
1. Собери проект: `npm run build` → `dist`.
2. Подключись к серверу по SSH (`ssh root@<IP>`).
3. **Не задень VPN:** проверь занятые порты (`ss -tlnp`). VPN обычно держит свой порт (часто 443 или кастомный) — выбери свободный для приложения (80, если свободен; иначе, например, 8080).
4. Поставь/настрой веб-сервер (nginx): отдаёт `dist`, со **SPA-fallback** `try_files $uri /index.html;` на выбранном порту.
5. Залей сборку на сервер (`scp -r dist/* root@<IP>:/var/www/app/` или `rsync`). `node_modules` не заливаем.
6. Открой порт в firewall (`ufw allow <порт>`), при необходимости — в панели провайдера.

### Фаза B3. Проверить и записать
1. Открой `http://<IP>:<порт>` — продукт открывается. Проверь внутренние роуты (прямой путь на внутренний экран не должен давать 404).
2. **Оговорки (сообщи пользователю):** адрес по `http` (без сертификата) — браузер покажет «не защищено», для теста/демо нормально; secure-context фичи (камера, геолокация, доступ к буферу) по голому IP не работают — обычному прототипу не нужны; красивый домен и HTTPS — отдельная настройка, здесь не делаем.
3. Запиши ссылку в `PROJECT.md` (Фаза финал).

## Фаза финал. Запись ссылки и итог
1. Создай/обнови `PROJECT.md`: живая ссылка (URL сервиса или `http://IP:порт`), путь деплоя (сервис/свой сервер), дата не нужна. Это вход для следующих шагов (метрики читают данные с живой версии).
2. **Доступы в `PROJECT.md` НЕ пиши** (ни пароль сервера, ни токен).
3. Выведи итог (см. ниже).

---

## Self-check (перед финалом)
- `npm run build` проходит; задеплоен свежий билд (не устаревший)?
- **SPA-маршрутизация работает вживую: прямой переход на внутренний экран не даёт 404?**
- Секреты/`.env`/`node_modules` НЕ в репозитории; пароль сервера и токены НЕ в файлах/`.git`/`PROJECT.md`?
- Свой сервер: выбран свободный порт, VPN не задет, порт открыт в firewall?
- Живая ссылка открывается и записана в `PROJECT.md`?

## Финал (вывести пользователю)
```
Продукт опубликован.
- Путь: <сервис (Vercel/Netlify/Cloudflare) | свой сервер по IP>
- Живая ссылка: <URL | http://IP:порт>
- SPA-маршрутизация: внутренние экраны открываются по прямой ссылке
- Записано в среду: PROJECT.md
Сохрани ссылку — по ней пойдёт юзер-тест и метрики.
```

## Грабли (учтено)
- **404 на внутренних экранах после деплоя** — главная грабля. SPA-приложение отдаёт один `index.html`; без fallback прямые ссылки на роуты ломаются. Сервис: `vercel.json`/`_redirects`; nginx: `try_files ... /index.html`.
- **На сервисе репозиторий устарел** (лежит только база с урока передачи) → деплоится не то. Перед импортом дозалить актуальный прототип (Фаза A1).
- **Ссылка Vercel просит логин** → по умолчанию включён Vercel Authentication (Deployment Protection). Выключить в Settings → Deployment Protection, иначе тестировщик не зайдёт.
- **Скопирована ссылка на конкретный билд** → она не обновляется на новые версии. Брать постоянную ссылку проекта (Production Domain).
- **Порт конфликтует с VPN** → приложение или VPN не поднимется. Проверить занятые порты, выбрать свободный.
- **Firewall закрыт** → по `http://IP:порт` не открывается. Открыть порт (ufw + панель провайдера).
- **`http` по IP** → secure-context фичи (камера/гео/буфер) не работают; «не защищено» — косметика. Для теста ок; нужен HTTPS — это домен + сертификат, отдельно.
- **Доступы в файлах** — пароль сервера/токен НЕ должны попасть в репозиторий, `.git/config`, `PROJECT.md`. Только для операции.
- **`node_modules` залит на сервер/в репо** — лишняя тяжесть. Заливаем `dist`, не исходники с зависимостями.

## Правила
- Два пути под одной директивой; путь выбирает пользователь в начале.
- SPA-маршрутизация обязательна в обоих путях (иначе внутренние экраны 404).
- Доступы (пароль сервера, токены) транзиентные — не в файлы.
- Свой сервер: открытие по `http://IP:порт`, без домена/HTTPS; не задевать VPN (свободный порт).
- Сервис: деплой из GitHub-репозитория, перед публикацией репозиторий обновляется. Свой сервер: заливка напрямую по SSH, GitHub не нужен.
- Живая ссылка пишется в `PROJECT.md`; повторный запуск обновляет публикацию тем же путём.
- Командную строку/подключение ведёт агент; ученик — браузерные шаги (сервис) или выдача доступа (свой сервер).
