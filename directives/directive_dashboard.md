# Директива: Подключить дашборд юзер-тестов (dashboard)

> **Запуск:** когда пользователь дал эту директиву и положил бандл `dashboard/` в проект. Переиспользуемая: первый запуск подключает, повторные — обновляют. Командную строку, сервер и подключение ведёт агент; ученик отдаёт доступы/токены, когда директива попросит, и проходит браузерные шаги.

## Задача
Подключить готовый дашборд юзер-тестов (бандл `dashboard/`) к текущему проекту: встроить трекер в прототип, поднять сбор событий на сервере, подтянуть сценарии и гипотезы из проекта, запустить дашборд. На выходе: живая версия прототипа шлёт поведение → сбор копит → дашборд показывает метрики и даёт hand-off агенту.

## Почему сбор живёт на твоём сервере (а не на Vercel)
Метрики — это сессии, клики и тайминги, которые копятся во времени; их принимает и хранит бэкенд. Сборщик из бандла это и есть: маленький сервер (Node, без зависимостей), который пишет события в файл. На Vercel/Netlify его не разместить — там статика и короткие функции без постоянного диска, накапливающийся файл там не живёт. Поэтому прототип может быть где угодно, а **данные всегда на твоём сервере** (том же VPS, что арендован под VPN). «Взрослую» управляемую БД не поднимаем — для юзер-тестов прототипа файлового хранилища достаточно, это сознательный выбор объёма.

## Когда применять
- После деплоя: прототип опубликован, есть живая ссылка. Тестируем именно живую версию.
- В проекте лежит распакованный бандл `dashboard/`.
- Переиспользуемая: добавились экраны/поведение → перезапуск обновляет трекер.

## Предусловие
- Бандл `dashboard/` в проекте (`tracker/`, `server/`, `app/`).
- **Свой сервер (VPS), Node 18+.** Доступ — IP / пользователь (root) / пароль, транзиентно. **Нет доступа — запроси у ученика** (это то, что выдал провайдер при аренде).
- Прототип задеплоен. Где именно (свой сервер или Vercel/сервис) определяет путь подключения — узнай из `PROJECT.md` (живая ссылка) или спроси ученика.

## Два пути подключения (выбор по тому, где живёт прототип)
Сбор всегда на сервере. Разница только в том, как трекер из прототипа достучится до сбора:
- **Путь 1 — прототип на своём сервере (`http://IP`):** всё на одной машине, один протокол. Трекер шлёт на `http://<IP>:<порт>`. Больше ничего не нужно.
- **Путь 2 — прототип на Vercel/сервисе (`https`):** HTTPS-страница не может слать на http-сервер — браузер блокирует это как mixed content. Мост — бесплатный туннель **ngrok**, который даёт сборщику https-адрес. Трекер шлёт на `https://<...>.ngrok-free.app`.

Определи путь по `PROJECT.md`. Не уверен — спроси: «прототип открывается по https (Vercel и т.п.) или по http://IP (свой сервер)?»

## Фаза 0. Контекст
1. Прочитай `dashboard/README.md` и `PROJECT.md` (живая ссылка = `prototypeUrl`, путь деплоя).
2. Возьми доступ к серверу (IP/root/пароль). Нет — запроси у ученика.
3. Выбери свободный порт сбора (например `8787`), не конфликтующий с приложением и VPN (`ss -tlnp`).
4. **Сценарии для config — из проекта, ученик не вписывает.** Источник: реестр экранов (например `src/screens/registry.ts`), роутер, PRD. Каждый сценарий → объект `flows[]`:
   - `id` — короткий слаг (`server-create`, `balance-topup`): метка U-теста в ссылке и точная привязка сессий.
   - `funnel` — ключевые экраны по порядку (для воронки и потерянности — длина пути = «оптимум»).
   - `goal` — целевой экран (= успех). По умолчанию последний шаг воронки.
   - `firstClick` (желательно) — стартовый экран и «верный» первый клик.
5. **Гипотезы — из проекта.** Источник: PRD (раздел гипотез/допущений), заметки. Каждая гипотеза = сценарий + **измеримый порог**, по нему дашборд выдаёт вердикт. Формат (строго как в `hypotheses.example.json`, иначе вердикта не будет):
   - `id`, `text` — идентификатор и формулировка.
   - `flow` — сценарий, к которому привязана (имя `name` из `flows[]`; слаг `id` тоже примет).
   - `measure` — что меряем: `completion` (доходимость до цели) / `firstclick` (верный первый клик) / `dropoff` (отвал на шаге) / `lostness` (потерянность).
   - `target` — порог: `{ "op": ">=" | "<=", "value": число }` (доля 0–1 для completion/firstclick/dropoff; для lostness — абсолют).
   - `step` — **только для `dropoff`**: экран шага, перед которым меряем отвал.
   - `impact`/`effort`/`status` — опц., для контекста.
   Без `flow`/`measure`/`target` гипотеза показывается, но вердикт = «недостаточно данных». Нет гипотез — пропусти, панель будет пустой.

## Фаза 1. Поднять сбор на сервере (оба пути)
1. Залей `dashboard/server/` на VPS (`scp`/`rsync`). `node_modules` не нужен — зависимостей нет.
2. Запусти фоновым процессом, чтобы пережил перезапуск: `pm2`, `systemd` или `nohup node server.js &`. Порт из Фазы 0 (`PORT=8787`). Хранилище данных задаётся `DATA_DIR` (по умолчанию `server/data`).
3. Проверь на сервере: `curl localhost:8787/health` → `{"ok":true}`.
4. (Опц.) Защити чтение метрик: запусти с `DASH_KEY=<секрет>` — дашборд тогда обращается с `?key=<секрет>`. Приём `/collect` всегда открыт. Если включаешь — ключ попадёт в `config.js`; **не коммить его** (в бандле уже есть `.gitignore` на `app/config.js`). Для прототип-теста защита необязательна, можно пропустить.

## Фаза 2. Дать сборщику адрес, который увидит прототип
### Путь 1 — прототип на своём сервере (http)
1. Открой порт сбора в firewall (`ufw allow 8787`) и при необходимости в панели провайдера.
2. Адрес сбора = `http://<IP>:8787`. Проверь снаружи: открой `http://<IP>:8787/health`.

### Путь 2 — прототип на Vercel (https) → туннель ngrok
1. **Запроси у ученика** (если ещё не дал): **ngrok authtoken** и его **постоянный адрес** (вида `xxx.ngrok-free.app`). Берутся в кабинете ngrok после бесплатной регистрации (разделы «Your Authtoken» и «Domains»).
2. На сервере поставь ngrok и привяжи токен: `ngrok config add-authtoken <token>`.
3. Подними туннель **фоновой службой** (systemd), чтобы пережил перезапуск, с явным постоянным адресом:
   `ngrok http 8787 --url=https://<его-адрес>`
4. Адрес сбора = `https://<его-адрес>`. Проверь: открой `https://<его-адрес>/health` → `{"ok":true}`.
   - Порт 8787 наружу в firewall открывать НЕ нужно — туннель идёт изнутри сервера.
   - Лимиты бесплатного ngrok (≈20k запросов/мес, 1 ГБ/мес) — для юзер-тестов с запасом.

## Фаза 3. Встроить трекер в прототип
1. Положи `dashboard/tracker/track.js` в статику прототипа (Vite — `public/track.js`).
2. Подключи на всех экранах — в корневой `index.html` перед `</body>`:
   `<script src="/track.js" data-endpoint="<АДРЕС СБОРА>/collect"></script>`
   Путь 1: `http://<IP>:8787/collect`. Путь 2: `https://<его-адрес>/collect`. Никогда не `localhost` — живую версию открывают с других устройств.
3. Трекер сам ловит переходы SPA, клики (с флагом интерактивности — для честного dead-клика), время и метку сценария `?ut_task=<id>`. Для ngrok бандл уже шлёт служебный заголовок — настраивать ничего не нужно. Качество находок растёт от разметки: проставь `data-track="..."` на ключевых кнопках/ссылках и на состояниях ошибок/пустых экранов (иначе ошибки в метрики не попадут — JS-исключения трекер не цепляет).

## Фаза 4. Конфиг и где открыть дашборд
1. Создай `dashboard/app/config.js` (по образцу `config.example.js`):
   ```js
   window.DASHBOARD_CONFIG = {
     endpoint: "<АДРЕС СБОРА>",          // Путь1: http://IP:8787 · Путь2: https://...ngrok-free.app
     prototypeUrl: "<живая ссылка>",      // реальный СТАРТОВЫЙ экран сценария, не дев-индекс
     screens: { "/route": "Имя экрана" }, // карта экранов (имена вместо слагов)
     flows: [ /* объекты сценариев из Фазы 0 */ ],
     key: "<DASH_KEY или пусто>"
   };
   ```
   `endpoint` обязателен и равен адресу сбора из Фазы 2 (это и есть закрытие mixed content: на Vercel-пути endpoint — https-туннель, протокол совпадает с прототипом).
   `prototypeUrl` должен вести на **реальный первый экран сценария**, куда приземляется респондент (напр. `/dashboard`), а не на служебный индекс/каталог экранов — проверь по роутеру, что корень `/` это не дев-список.
2. **Где открыть дашборд** (это статика, читает сбор по `endpoint`):
   - **Путь 1:** положи `dashboard/app/` на сервер отдельной раздачей (nginx, отдельный путь/порт) — откроешь по своей ссылке, например `http://<IP>/dashboard`. Так и приложение, и дашборд на одной машине, по разным ссылкам.
   - **Путь 2:** так же на сервер, либо открой локально у себя (он читает сбор через https-туннель). Респондентам дашборд не нужен — они открывают прототип.
3. Дашборд читает `config.js` с антикэшем; кнопка «Обновить» перечитывает данные во время теста.

## Фаза 5. Гипотезы на сервер (если есть)
- Отправь JSON из Фазы 0: `POST <АДРЕС СБОРА>/hypotheses`, либо положи файл `data/hypotheses.json` рядом со сборщиком. Дашборд их подхватит и оценит вердикт по результату теста.

## Фаза 6. Связать, проверить, зафиксировать раунд
1. Передеплой прототип с трекером (директивой деплоя): Путь 1 — перезалить `dist` на сервер; Путь 2 — пуш в репозиторий, Vercel пересоберёт. **Что деплой докатился — смотри по статусу коммита на GitHub** (Vercel постит туда результат: success/failed) или подожди ~1–2 мин. **Не опрашивай живой URL частым `curl`** — Vercel примет это за бота, включит Security Checkpoint (403) и заблокирует сайт для всех, включая респондентов.
2. Открой дашборд → вкладка «Сценарии» → возьми **ссылку для U-теста** (`prototypeUrl?ut_task=<id>`), пройди по ней сценарий сам и раздай респондентам. Всё, что пройдут по ней, привяжется к этому сценарию по метке.
3. «Обновить» — появятся сессии, воронка с контекстом выхода, friction по экранам, first-click, потерянность, авто-находки.
4. **Зафиксируй раунд** — снимок метрик по всем сценариям. После правок агента и ретеста блок «Цикл итерации» покажет было→стало.

## Фаза 7. Бриф агенту и цикл
- В дашборде «Все находки → задание» (или по одной находке) → вставляешь в Claude Code → агент предлагает и вносит правки → передеплой → ретест → «Обновить». По критериям и блоку было→стало видно, помогла ли правка.

## Фаза 8. Повторный запуск
- Новые экраны/поведение → передеплой прототип (трекер уже подключён). Сбор и туннель не трогаешь.
- Сменился адрес сбора → поправь `data-endpoint` в прототипе и `endpoint` в `config.js`.

## Грабли (учтено)
- **Прототип на https (Vercel), сбор на `http://IP` → браузер блокирует (mixed content), события не доходят.** Это главная грабля Vercel-пути. Решение — Путь 2: туннель ngrok даёт сборщику https-адрес, протоколы совпадают. Не пытайся слать с https-страницы на `http://IP`.
- **ngrok без постоянного адреса** → при запуске без `--url=<твой-адрес>` ngrok выдаёт случайный адрес, меняющийся при перезапуске и рвущий тест. Бери постоянный адрес из кабинета и запускай с `--url=`.
- **Регистрация в ngrok через GitHub не проходит на российской почте** — если GitHub-аккаунт привязан к mail.ru (и подобным), ngrok не пустит регистрацию через GitHub. Регистрируйся через Google/Gmail-аккаунт, дальше разницы нет.
- **Туннель/сбор подняты «на время сессии»** (в обычном терминале) → закрыл — умерло. Поднимай службой (systemd/pm2), не в терминале.
- **`data-endpoint` = `localhost`** → с других устройств события не уходят. Только публичный адрес сбора.
- **Порт сбора закрыт/занят/конфликтует с VPN** (Путь 1) → `/collect` недоступен. Открыть порт, выбрать свободный.
- **CORS** — сборщик уже шлёт `Access-Control-Allow-Origin: *` и разрешает служебный заголовок ngrok; правок не нужно.
- **Два сценария с одного стартового экрана / респондент гуляет между флоу** → привязка «по входу» их смешает. Решение — метка: раздавай ссылку U-теста `prototypeUrl?ut_task=<id>`, сессии привяжутся к сценарию точно (в дашборде статус «привязка по метке»).
- **Доступы/токены в файлах** — пароль сервера и ngrok-токен только для операции, не в репозиторий/`.git`/`PROJECT.md`.
- **Ученик ищет, куда вводить адрес** — никуда, полей в дашборде нет. Всё в `config.js`, который генерит директива. «Не подключён» → `config.js` не создан или `endpoint` пуст (Фаза 4).
- **Vercel Security Checkpoint (403 `x-vercel-mitigated: challenge`)** — частый `curl`-поллинг живого URL включает бот-защиту Vercel, и сайт начинает отдавать 403 всем. Не долби URL автопроверками; статус деплоя бери из коммит-статуса на GitHub. Если уже словил — отпустит само за ~10–30 мин, или в Vercel → проект → Firewall выключи Attack Challenge Mode.
- **Vercel блокирует деплой по «Git Author»** — если коммит подписан email'ом, не привязанным к GitHub-аккаунту, Vercel не соберёт. Коммить email'ом аккаунта (часто это `…@users.noreply.github.com`); проверь `git config user.email` в папке прототипа.
- **`prototypeUrl` ведёт на дев-индекс/каталог экранов** (корень `/` бывает служебным списком) → респондент приземляется не туда, метка сценария и воронка сбиваются. Ставь `prototypeUrl` на реальный первый экран сценария (сверься с роутером).

## Правила
- Сбор (бэкенд + хранилище) всегда на своём сервере; на Vercel только фронт (прототип, при желании статика дашборда).
- Путь выбирается по тому, где прототип: свой сервер (http) или Vercel (https → туннель ngrok).
- Доступы и токены транзиентные — не в файлы/репозиторий.
- Подключение целиком в `config.js` + `data-endpoint`; ученик руками в дашборде ничего не вводит.
- Трекер собирает только поведение в прототипе, без персональных данных и ввода.
- Повторный запуск обновляет трекер; hand-off находок агенту замыкает круг с итерацией прототипа.
