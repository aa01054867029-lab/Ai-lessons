# Директива: Оживление — навигация, данные, поведение (wire)

> **Запуск:** только когда пользователь явно дал эту директиву. **Переиспользуемая, стоячий инструмент:** возвращаешься к ней весь проект — оживить ещё экран, добавить поведение, доанимировать. Повторный запуск наращивает, не ломает уже подключённое.

## Задача
Превратить статичные экраны (из `directive_screens`) в рабочее приложение: подключить навигацию между экранами, слой мок-данных, обработку форм и состояния по действию — включая краевые (edge-cases). Designer задаёт логику (что по какому действию), агент пишет обработчики. На выходе — кликабельный продукт, по которому можно ходить и тестировать.

## Когда применять
- После `directive_screens`: экраны собраны и зарегистрированы в реестре, но статичны (без логики и данных).
- **На любом шаге и многократно:** оживить первый экран, потом другой; добавить новое поведение; позже — анимации. Каждый запуск дополняет. Это не разовый прогон.

## Предусловие
- Собраны статичные экраны и есть реестр/роутер (`src/screens/`), база `src/components/`, токены `src/tokens/`.
- Прочитать `ds/CONTRACT.md` (правила, токены состояний) и `ds/components.md` (какие компоненты под состояния — Toast, Badge, пустые состояния и т.д.).

## Ключевой принцип
Статичные экраны становятся живыми за счёт четырёх вещей: **навигация** (переходы между экранами реестра), **данные** (один слой мок-данных, который экраны читают и меняют), **формы** (ввод → действие → результат), **состояния по действию** (загрузка / пусто / ошибка / успех + edge-cases). Логику задаёт пользователь, обработчики пишет агент — в код пользователь не лезет. Данные — **мок**, без настоящего бэкенда. Всё на семантических токенах (`CONTRACT`).

## Граница (что НЕ делаем)
- **Не подключаем настоящий бэкенд / API / базу.** Только мок-слой данных в коде. Реальные данные — позже, отдельно.
- **Не деплоим** (хостинг — отдельная директива) и **не ставим метрики/трекинг** (дашборд — отдельная директива). Здесь только поведение приложения на моке.
- Не переписываем базу компонентов и не пересобираем экраны с нуля — берём готовое и оживляем.

## Фаза 0. Контекст
Посмотри реестр экранов (`src/screens/registry.*`) и сами экраны — что собрано, какой контент сейчас статикой. Прочитай `CONTRACT` и `components.md` (компоненты под состояния). Определи с пользователем, какой экран/сценарий оживляем в этот запуск (всё сразу не обязательно — можно по одному).

## Фаза 1. Навигация
Сделай навигацию рабочей: клики по пунктам меню, кнопкам и ссылкам ведут на нужные экраны реестра; работает переход «назад» и история; активный пункт меню подсвечивается. Экраны уже зарегистрированы — здесь они становятся реально проходимыми как продукт.

## Фаза 2. Слой мок-данных
1. Заведи **один** слой данных (`src/data/`) — мок-данные, которые нужны экранам (например, список серверов, баланс, операции). Реалистичные, но это мок, не бэкенд.
2. Переведи экраны со статичного контента (вшитого в урок сборки) на чтение из этого слоя. Действия смогут его менять (создал сервер → запись добавилась → список обновился). Это и есть превращение «картинки с текстом» в данные.

## Фаза 3. Формы и поведение
1. Оживи формы: поля собираются, на отправку — действие (добавить запись, изменить данные), с результатом на экране.
2. Валидация ввода: обязательные поля, форматы, границы (пустое, слишком длинное, неверная сумма) → состояние ошибки на поле.
3. **Конкретное поведение задаёт пользователь.** Очевидное из экранов агент вешает сам (submit формы, переход по кнопке, создание → список). Для продуктовой логики, которой нет в макете, — спроси пользователя, что по какому действию должно происходить, и реализуй. Логику формулирует он, пишешь ты.

## Фаза 4. Состояния и edge-cases
Состояния по действию — обязательная часть, не только «когда всё хорошо». Заложи стандартный набор и покажи на экранах:
- **загрузка** (действие в процессе), **успех** (подтверждение, Toast), **ошибка** (действие не прошло), **пусто** (список/данные пустые — empty state);
- **edge-cases**: невалидный ввод, граничные значения (ноль, минус, слишком много/длинно), доменные края (нулевой/низкий баланс → баннер + блокировка действия), «ничего не найдено».
Все состояния — из компонентов базы и семантических токенов (Toast, Badge, баннеры, пустые состояния по `components.md`). **По продукту предложи дополнительные edge-cases**, которых нет в макете, — пользователь решает, какие добавить.

## Фаза 5. Анимации и переходы (опционально, итеративно)
По желанию пользователя — оживить движением: плавные переходы между экранами, появление/исчезновение элементов (добавление в список, тосты, модалки), микро-взаимодействия. Базовые, аккуратные (≈150–250ms, ease). Это можно сделать сейчас или отдельным повторным запуском позже — добавляется поверх, не ломая поведение.

## Фаза 6. Прокликать и проверить
1. `npm run dev`, пройди оживлённый сценарий целиком, как пользователь.
2. Ищи дыры: тупик, откуда не вернуться; действие без реакции; состояние, которое не предусмотрено; кривой переход. Нашёл — правишь через агента, не руками.
3. Покажи пользователю, что подключено; расхождения он называет словами.

## Фаза 7. Повторный запуск = наращивание
Стоячий инструмент. При повторном запуске:
- **ещё экран/сценарий** — оживи его по тем же фазам, остальное не трогай;
- **ещё поведение** — добавь конкретную логику по запросу пользователя;
- **анимации** — добавь движение поверх (Фаза 5).
Существующее не ломаем, мок-слой и навигацию переиспользуем, дубликаты не плодим. В финале перечисли, что добавил.

## Self-check (перед финалом)
- Навигация работает (переходы, «назад», активный пункт)?
- Экраны читают данные из единого мок-слоя, действия его меняют (а не статичный текст)?
- Формы валидируются, на отправку — результат?
- Заложены состояния: загрузка / успех / ошибка / пусто + edge-cases (границы, нулевой баланс, «ничего не найдено»)?
- Всё на компонентах базы и семантических токенах; прибитых значений нет?
- Не залезли в бэкенд / деплой / метрики (это отдельные директивы)?
- `npm run dev` без ошибок в консоли.

## Финал (вывести пользователю)
```
Оживлено.
- Навигация: <что заработало>
- Мок-данные: src/data/ — экраны читают и меняют
- Формы: <какие> с валидацией
- Состояния: загрузка/успех/ошибка/пусто + edge-cases (<перечень>)
- Анимации: <добавлены / предложены / нет>
Оживлено в этот запуск: <экраны/сценарии>. Осталось статичным: <список или «нет»>
Бэкенд/деплой/метрики не трогал — отдельные шаги.
```

## Грабли (учтено)
- **Агент тянет настоящий бэкенд/API** — на этом шаге только мок-слой; реальные данные отдельно. Иначе разрастётся не туда.
- **Контент остаётся статикой** — если экраны не перевели на слой данных, действия ничего не меняют. Перевод статики → данные обязателен (Фаза 2).
- **Happy-path-only** — частая недоработка: оживили «когда всё хорошо», забыли пусто/ошибку/границы. Edge-cases — обязательная часть (Фаза 4), не опция.
- **Прибитые значения в новых состояниях** — состояния тоже на семантических токенах и компонентах базы.
- **Повторный запуск ломает сделанное** — наращиваем, переиспользуем мок-слой/навигацию, существующее не переписываем (Фаза 7).
- **Заехали в деплой/метрики** — это следующие уроки; здесь только поведение на моке.

## Правила
- Оживляем готовые экраны; базу и экраны с нуля не пересобираем.
- Данные — мок, без бэкенда. Деплой и метрики — отдельные директивы.
- Логику задаёт пользователь — агент пишет обработчики; пользователь в код не лезет.
- Edge-cases — стандарт, не опция; дополнительные предлагаем по продукту.
- Только семантические токены и компоненты базы (`CONTRACT`).
- Переиспользуемая: повторный запуск наращивает (ещё экран, поведение, анимации), не ломает.
